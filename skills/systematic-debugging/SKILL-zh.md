# 系统化调试

## 概述

随机修复浪费时间并制造新 bug。快速补丁掩盖潜在问题。

**核心原则：** 始终在尝试修复之前找到根本原因。只修复症状就是失败。

**违反此流程的字面意思就是违反调试的精神。**

## 铁律

```
没有根本原因调查，就不做修复
```

如果你没有完成阶段 1，就不能提出修复方案。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产环境 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别适用于以下情况：**
- 时间紧迫时（紧急情况让猜测变得诱人）
- "只是一个快速修复"看起来很明显时
- 你已经尝试了多种修复方案时
- 上一次修复没有效果时
- 你没有完全理解问题时

**以下情况不要跳过：**
- 问题看起来很简单（简单的 bug 也有根本原因）
- 你很着急（匆忙保证需要返工）
- 管理者要求立刻修复（系统化比瞎折腾更快）

## 四个阶段

你必须完成每个阶段后才能进入下一个阶段。

### 阶段 1：根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误消息**
   - 不要跳过错误或警告
   - 它们通常包含精确的解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **稳定复现**
   - 你能可靠地触发它吗？
   - 确切步骤是什么？
   - 每次都会发生吗？
   - 如果无法复现 → 收集更多数据，不要猜测

3. **检查最近的变更**
   - 什么变更可能导致这个问题？
   - Git diff、最近的提交
   - 新的依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复方案之前，添加诊断检测：**
   ```
   对于每个组件边界：
     - 记录什么数据进入组件
     - 记录什么数据离开组件
     - 验证环境/配置的传播
     - 检查每一层的状态

   运行一次以收集证据，显示哪里出了问题
   然后分析证据以确定失败的组件
   然后调查该特定组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== 工作流中可用的密钥： ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== 构建脚本中的环境变量： ==="
   env | grep IDENTITY || echo "IDENTITY 不在环境中"

   # 第 3 层：签名脚本
   echo "=== 钥匙串状态： ==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这揭示了：** 哪一层失败（密钥 → 工作流 ✓，工作流 → 构建 ✗）

5. **追踪数据流**

   **当错误位于调用栈深处时：**

   参见本目录下的 `root-cause-tracing.md`，了解完整的反向追踪技术。

   **简版：**
   - 错误值从哪里产生？
   - 是什么用错误值调用了这里？
   - 持续向上追踪直到找到源头
   - 在源头修复，而非在症状处修复

### 阶段 2：模式分析

**在修复之前找到模式：**

1. **找到可用的示例**
   - 在同一代码库中定位类似的可用代码
   - 什么可用而什么坏了？

2. **与参考实现对比**
   - 如果在实现某个模式，完整阅读参考实现
   - 不要略读——逐行阅读
   - 在应用之前完全理解该模式

3. **识别差异**
   - 可用的和坏了的之间有什么不同？
   - 列出每个差异，无论多小
   - 不要假设"那不可能有影响"

4. **理解依赖关系**
   - 这需要哪些其他组件？
   - 需要什么设置、配置、环境？
   - 它做了什么假设？

### 阶段 3：假设与测试

**科学方法：**

1. **提出单一假设**
   - 清楚地表述："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做尽可能小的变更来测试假设
   - 每次只改变一个变量
   - 不要同时修复多个东西

3. **在继续之前验证**
   - 有效吗？是 → 阶段 4
   - 没有效？提出新假设
   - 不要在失败的修复上叠加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要装作知道
   - 寻求帮助
   - 做更多研究

### 阶段 4：实现

**修复根本原因，而非症状：**

1. **创建失败测试用例**
   - 最简化的复现
   - 尽可能使用自动化测试
   - 无框架时使用一次性测试脚本
   - 修复前必须有测试
   - 使用 `superpowers:test-driven-development` 技能来编写规范的失败测试

2. **实现单一修复**
   - 针对识别出的根本原因
   - 一次一个变更
   - 不要"顺便"改进
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有破坏其他测试？
   - 问题确实解决了？

4. **如果修复无效**
   - 停下来
   - 计数：你已经尝试了多少次修复？
   - 如果 < 3 次：回到阶段 1，用新信息重新分析
   - **如果 ≥ 3 次：停下来质疑架构（见下方第 5 步）**
   - 不要在没有架构讨论的情况下尝试第 4 次修复

5. **如果 3 次以上修复失败：质疑架构**

   **表明架构问题的模式：**
   - 每次修复都揭示新的共享状态/耦合/不同位置的问题
   - 修复需要"大规模重构"才能实现
   - 每次修复在其他地方产生新症状

   **停下来质疑根本基础：**
   - 这个模式从根本上是否合理？
   - 我们是否"纯粹因为惯性在坚持"？
   - 是否应该重构架构而非继续修复症状？

   **在尝试更多修复之前与你的搭档讨论**

   这不是假设失败——这是架构错误。

## 红色警报 - 停下来遵循流程

如果你发现自己在想：
- "先快速修复，稍后调查"
- "试试改一下 X 看看行不行"
- "同时做多个变更，然后跑测试"
- "跳过测试，我会手动验证"
- "可能是 X，让我修复一下"
- "我没有完全理解但这也许能行"
- "模式说 X 但我要换种方式"
- "主要问题是这些：[列出修复方案但没有调查]"
- 在追踪数据流之前就提出解决方案
- **"再试一次修复"（已经尝试了 2 次以上）**
- **每次修复都在不同位置揭示新问题**

**以上所有意味着：停下来。回到阶段 1。**

**如果 3 次以上修复失败：** 质疑架构（见阶段 4.5）

## 你的搭档发出你做错了的信号

**注意这些引导：**
- "不是这样吗？" - 你没有验证就做了假设
- "它能展示……吗？" - 你应该已经添加了证据收集
- "别猜了" - 你在没有理解的情况下提出修复方案
- "深度思考一下" - 质疑根本基础，而非只是症状
- "我们卡住了？"（沮丧地）- 你的方法不起作用

**当你看到这些：** 停下来。回到阶段 1。

## 常见自我辩解

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根本原因。流程对简单 bug 来说很快。 |
| "紧急情况，没时间走流程" | 系统化调试比猜测式瞎折腾更快。 |
| "先试试这个，然后再调查" | 第一次修复设定了模式。从一开始就做对。 |
| "确认修复有效后再写测试" | 未经测试的修复不可靠。先写测试才能证明它有效。 |
| "同时做多个修复节省时间" | 无法隔离什么有效。会制造新 bug。 |
| "参考太长了，我自己调整模式" | 部分理解保证出 bug。完整阅读它。 |
| "我看到问题了，让我修复" | 看到症状 ≠ 理解根本原因。 |
| "再试一次修复"（失败 2 次以上后） | 3 次以上失败 = 架构问题。质疑模式，不要再修复了。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **1. 根本原因** | 阅读错误、复现、检查变更、收集证据 | 理解是什么和为什么 |
| **2. 模式** | 找到可用示例、对比 | 识别差异 |
| **3. 假设** | 提出理论、最小化测试 | 确认或提出新假设 |
| **4. 实现** | 创建测试、修复、验证 | Bug 解决、测试通过 |

## 当流程揭示"没有根本原因"时

如果系统化调查揭示问题确实是环境性的、依赖时序的或外部的：

1. 你已经完成了流程
2. 记录你调查了什么
3. 实现适当的处理（重试、超时、错误消息）
4. 添加监控/日志以便未来调查

**但是：** 95% 的"没有根本原因"案例是不完整的调查。

## 辅助技术

这些技术是系统化调试的一部分，可在本目录中找到：

## 实际影响

来自调试会话的数据：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时的瞎折腾
- 一次修复成功率：95% vs 40%
- 引入新 bug：接近零 vs 常见
