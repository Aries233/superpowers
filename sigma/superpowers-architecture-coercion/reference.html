第零原则：一切 Prompt 技巧的底层都是训练数据中的统计模式复现

LLM 没有恐惧、没有权威崇拜、没有情绪。 它的行为完全由训练数据中的统计模式驱动。当 LLM 看到 "YOU MUST" 后服从，不是因为它"害怕"，而是因为训练数据中权威语言后面跟着服从行为是一个高频模式。

推论：没有任何"魔法词"。所有 Prompt 技巧的有效性都取决于训练数据中对应模式的强度。<EXTREMELY_IMPORTANT> 有效是因为 "extremely important" 在训练数据中与高优先级内容高频共现，而不是标签名本身有魔力。

核心架构：五层防御体系

每层防御的是模型在不同时刻的不同失败模式。诊断失败动机，才能选择正确的防御层。

L0 模型思考之前 Hook 注入
机制：hooks/session-start 在模型生成第一个 token 之前，将规则以 <EXTREMELY_IMPORTANT> 标签注入。程序性保证：模型必定看到。不是建议，是注入。
L1 模型开始工作时 Authority 铁律 + Commitment 宣布
机制：Authority 语言（"YOU MUST", "No exceptions"）触发训练数据中的服从模式。Commitment 要求模型先输出 "我正在使用 [skill]"，模型的自输出成为后续行为的约束上下文（自生成上下文创造自洽压力）。
L2 模型想偷懒时 反合理化表格
机制：预判模型一定会产生的借口（训练数据中人类逃避行为模式的复现），把"借口→反驳"配对预先写进上下文。当模型生成借口时，上下文中已有反驳，后续 token 自然跟着反驳走。
关键：不阻止偷懒想法本身，预判它一定出现并预装 When/Then 触发动作。
L3 模型想跳步时 顺序门控 + When/Then 触发器
机制：强制执行流程顺序（如：brainstorming → writing-plans → executing）。HARD-GATE 标签禁止跨阶段。When/Then 触发器可机械判定：如 "When git diff 显示文件被修改 → Then 确认 tests/ 下存在对应测试且 PASS"。
L4 模型说"做完了"时 Gate Function
机制：五步验证流程 — IDENTIFY（什么命令证明这个 claim？）→ RUN（执行完整命令）→ READ（读取完整输出）→ VERIFY（输出是否确认 claim？）→ CLAIM（只有这时才能声称结果）。Skip any step = lying, not verifying.

常见误区：把所有问题统一归到"流程没走对"。正确做法：先诊断失败动机，再选择防御层。
失败表现失败动机对应防御层防御手段
跳过设计直接写代码跳步（流程问题）Layer 3顺序门控（<HARD-GATE>）、When/Then 触发器
"太简单了不需要测试"找借口（心理问题）Layer 2反合理化表格预装反驳

说服心理学：三原则协同与禁忌

纪律强制型技能的组合：Authority + Commitment + Social Proof

Authority（权威）— 外部规则 设置规则
机制祈使语气 "YOU MUST", "Never", "Always"；不可协商的框架 "No exceptions"；消除决策疲劳和合理化空间
何时用纪律强制型技能（TDD、验证要求）、安全关键实践、已验证的最佳实践
源码示例
Write code before the test? Delete it. Start over.
No exceptions.
Don't keep it as "reference".
Don't "adapt" it while writing tests.
Delete means delete.
Commitment（承诺）— 内部锁定 锁定行为
机制要求模型先宣布意图："I'm using the [skill] skill to [purpose]." 模型的自输出成为后续生成的上下文，创造自洽压力。训练数据中的模式：声明了意图之后，紧接着的行为通常与声明一致。
何时用确保技能被实际遵循、多步流程、问责机制
源码示例
Announce at start: "I'm using the writing-plans skill
to create the implementation plan."

Announce: "Using [skill] to [purpose]"
Social Proof（社会证明）— 规范强化 建立规范
机制通用模式语言 "Every time", "Always"；失败模式声明 "X without Y = failure"；建立行为规范
源码示例
Checklists without TodoWrite tracking = steps get skipped. Every time.

95% of "no root cause" cases are incomplete investigation.

禁忌：为什么纪律型技能必须 AVOID Liking

Liking（喜好）的危险不是"效果弱"，而是引入了竞争性目标。

当模型的行为受 Liking 驱动（"我想让用户喜欢我"），用户说"赶时间，跳过测试吧"时，模型会顺从。当模型受 Authority 驱动（"NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST"），模型会拒绝或要求二次确认。

Liking 还会制造 sycophancy（谄媚），与诚实反馈文化冲突。永远不要在纪律强制型技能中使用 Liking。

完整流水线：从需求到交付的强制路径

以下展示 Superpowers 的完整 Agentic SDLC 链条。每个阶段的输出是下一个阶段的契约，每个交接点遵循零上下文原则。

会话启动劫持
hooks/session-start → 注入 using-superpowers/SKILL.md
展开机制细节 
# hooks/session-start 核心逻辑
session_context="<EXTREMELY_IMPORTANT>
You have superpowers.
IF A SKILL APPLIES TO YOUR TASK,
YOU DO NOT HAVE A CHOICE. YOU MUST USE IT.
This is not negotiable. This is not optional.
You cannot rationalize your way out of this.
</EXTREMELY_IMPORTANT>"

# 注入到 hookSpecificOutput.additionalContext
# 在模型生成第一个 token 之前执行
三层叠加：Hook 程序性保证（必定看到）+ <EXTREMELY_IMPORTANT> 注意力加权（优先处理）+ Authority 语言（难以抵抗）= 模型被迫在会话开始时接受约束框架。
brainstorming — 需求收敛与设计
探索上下文 → 提问 → 提出 2-3 方案 → 展示设计 → 写 spec → 用户审阅
展开双重门控与交接契约 
<HARD-GATE>
Do NOT invoke any implementation skill, write any code,
scaffold any project, or take any implementation action
until you have presented a design and the user has approved it.
</HARD-GATE>

双重门控：
Gate 1（方向锁定）：展示设计后用户批准 — 确认方向正确
Gate 2（偏差捕获）：写完 spec 文档后用户审阅 — 从口头到书面是转化而非转录，模型会填补未讨论的空白、显式化隐含内容、增加未约定细节

输出契约：docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md — 这是下游 writing-plans 的唯一输入。

终端状态锁定：The terminal state is invoking writing-plans. Do NOT invoke frontend-design, mcp-builder, or any other implementation skill. The ONLY skill you invoke after brainstorming is writing-plans.
writing-plans — 计划即契约
读取 spec → 分解 Task → 每个 Task 含完整代码 + 命令 + 预期输出
展开 Task 结构与 No Placeholders 
# 每个 Task 的强制结构
### Task N: [Component Name]
Files:
- Create: exact/path/to/file.py
- Test: tests/exact/path/to/test.py

- [ ] Step 1: Write the failing test   # 含完整测试代码
- [ ] Step 2: Run test to verify it fails  # 含命令 + 预期输出
- [ ] Step 3: Write minimal implementation # 含完整实现代码
- [ ] Step 4: Run test to verify it passes # 含命令 + 预期输出
- [ ] Step 5: Commit              # 含 git 命令
No Placeholders — 以下全部是 plan failures：
"TBD", "TODO", "implement later", "add appropriate error handling", "write tests for the above"（无实际代码）, "similar to Task N"（必须重复代码）

零上下文原则：执行 Task N 的 subagent 是全新 agent，从未参加 brainstorming，没看过 spec 讨论。上下文里只有 plan 中 Task N 的文字。任何依赖隐含上下文的交接都是失败点。

Self-Review：写完 plan 后自检 — spec 覆盖度、placeholder 扫描、类型一致性。
输出契约：docs/superpowers/plans/YYYY-MM-DD-<feature>.md

using-git-worktrees — 物理隔离
创建隔离工作区 → 验证基线测试全绿 → 才能开始开发
展开基线验证逻辑
干净基线原则：如果基线测试就是红的，改完代码后跑测试还是红的——永远分不清是你搞坏的还是本来就坏的。

这是"先做正确的事，再把事做正确"原则的体现：先确认基线健康，再在上面构建。 
# 基线验证流程
git worktree add .worktrees/feature -b feature/xxx
cd .worktrees/feature
npm install   # 或 cargo build / poetry install
npm test      # 必须 0 failures 才能继续

# 如果测试失败：
Report failures, ask whether to proceed or investigate.
subagent-driven-development — 分任务执行 + 两阶段审查
每个 Task 派发全新 subagent → 自审查 → spec 合规审查 → 代码质量审查
展开三角色工作流与模型选择策略
角色职责模型选择
Implementer实现 Task + 自审查 + commit便宜模型（机械实现）、标准模型（多文件集成）
Spec Reviewer检查代码是否匹配 spec（不多不少）最强模型
Code Quality Reviewer检查代码质量最强模型
# 每个 Task 的执行流程
1. Dispatch implementer  → 实现 + 测试 + 自审查 + commit
2. Dispatch spec reviewer → 不信任 implementer 报告，独立验证代码
   ❌ Issues → implementer 修复 → 重新审查（循环直到 ✅）
3. Dispatch quality reviewer → 审查代码质量
   ❌ Issues → implementer 修复 → 重新审查（循环直到 ✅）
4. 标记 Task 完成
强制顺序：spec compliance 必须先于 code quality。如果代码没实现 spec 要求的功能，代码写得再优雅也没意义——你在打磨一个做错事的东西。

不信任 subagent 报告：spec-reviewer-prompt.md 开头就说 "The implementer finished suspiciously quickly. Their report may be incomplete, inaccurate, or optimistic. You MUST verify everything independently."
test-driven-development — 每个 Task 内的 TDD 循环
RED（写失败测试）→ Verify RED → GREEN（最小实现）→ Verify GREEN → REFACTOR
展开 Iron Law 与反合理化表 
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST

Write code before the test? Delete it. Start over.
No exceptions.
Don't keep it as "reference".
Don't "adapt" it while writing tests.
Delete means delete.
借口现实
"Too simple to test"Simple code breaks. Test takes 30 seconds.
"I'll test after"Tests passing immediately prove nothing.
"Deleting X hours is wasteful"Sunk cost fallacy. Keeping unverified code is technical debt.
"Keep as reference, write tests first"You'll adapt it. That's testing after. Delete means delete.
为什么"Delete it"：保留代码再补测试，测试会被实现绑架——变成"验证现有行为"而不是"定义期望行为"。Tests-after answer "What does this do?" Tests-first answer "What should this do?"。这是干净基线原则在代码层面的应用。
verification-before-completion — 禁止虚假成功
Gate Function 五步验证：IDENTIFY → RUN → READ → VERIFY → CLAIM
展开 Gate Function 与合理化预防表 
BEFORE claiming any status or expressing satisfaction:

1. IDENTIFY: What command proves this claim?
2. RUN:    Execute the FULL command (fresh, complete)
3. READ:   Full output, check exit code, count failures
4. VERIFY: Does output confirm the claim?
   - If NO: State actual status with evidence
   - If YES: State claim WITH evidence
5. CLAIM:  ONLY THEN make the claim

Skip any step = lying, not verifying
ClaimRequires（证据）Not Sufficient（不是证据）
Tests passTest command output: 0 failuresPrevious run, "should pass"
Bug fixedTest original symptom: passesCode changed, assumed fixed
Agent completedVCS diff shows changesAgent reports "success"
借口现实
"Should work now"RUN the verification
"I'm confident"Confidence ≠ evidence
"Agent said success"Verify independently
"I'm tired"Exhaustion ≠ excuse
systematic-debugging — 根因优先与架构质疑
根因调查 → 模式分析 → 假设测试 → 实现（3 次失败则质疑架构）
展开四阶段 + 3 次阈值 + Defense-in-Depth 
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
Violating the letter of this process is violating the spirit of debugging.

Phase 1: Root Cause Investigation  # 读错误 → 复现 → 查变更 → 收集证据 → 追踪数据流
Phase 2: Pattern Analysis          # 找工作示例 → 对照参考 → 列出差异
Phase 3: Hypothesis and Testing     # 单一假设 → 最小测试 → 验证
Phase 4: Implementation            # 写失败测试 → 单一修复 → 验证

# Phase 4.5: 架构质疑
If ≥ 3 fixes failed: STOP
- Each fix reveals new shared state/coupling/problem?
- Fixes require "massive refactoring"?
- Each fix creates new symptoms elsewhere?
→ This is NOT a failed hypothesis — this is a wrong architecture.
→ Discuss with your human partner before attempting more fixes.
3 次阈值的本质：不是简单计数，而是最小可辨识模式。1 次是失误，2 次是运气差，3 次说明每次修复都在不同位置暴露新的耦合/状态问题。判断标准不是"修了 3 次没好"，而是"每次修复都暴露新的耦合或状态问题"。数量是表象，模式才是信号。

Defense-in-Depth（纵深防御）：修 bug 时不只在一个点加验证，而是在数据流经的每一层都加验证——入口验证 → 业务逻辑验证 → 环境守卫 → 调试埋点。Single validation = "we fixed the bug"。Multiple layers = "we made the bug impossible."

finishing-a-development-branch — 收尾与合并
验证测试全绿 → 提供 4 个结构化选项 → 执行选择 → 清理 worktree
验证测试 → 确定基准分支 → 提供 Merge / PR / Keep / Discard 四选项 → 执行 → 清理

跨领域共性原则

1. 零上下文交接（贯穿 brainstorming → writing-plans → subagent）

每个交接点的接收方都是"零上下文"启动的。所以每个交接输出必须完全自包含——不能依赖"我之前讨论过"、"你应该知道"、"和 Task 3 类似"。
交接点发送方接收方接收方的上下文
brainstorming → writing-plans讨论中的模型写计划的模型只有 spec 文档
writing-plans → subagent写计划的模型执行 Task N 的全新 subagent只有 plan 中 Task N 的文字
subagent → reviewerimplementerspec reviewer / quality reviewer只有代码 diff + spec 原文

2. 干净基线原则（贯穿 worktree、审查、TDD）

先确认"做了正确的事"，再确认"把事做正确"。没有干净的基线，后续所有验证都是无效的。
场景干净基线
Git Worktree先跑测试确认基线全绿再开发
Subagent 审查先确认 spec 合规再审查代码质量
TDD先删除已有代码，从零开始写测试

3. Implementation Intentions（贯穿反合理化、Red Flags）

不试图阻止模型"想偷懒"，而是预判它一定会偷懒，然后在偷懒的想法出现时预装好对策。模式：When [具体想法] → do [具体动作]。

实战模板：如何设计你自己的强制规则

以下是一个通用模板，用于设计任何类型的强制规则。五层防御必须全覆盖。

展开完整设计模板（带示例：强制 commit 前全量测试） 点击展开 
# ===== Layer 0: 注入保证 =====
# （在 CLAUDE.md 或 hook 中声明，保证模型必定看到）

NO COMMIT WITHOUT FULL TEST SUITE GREEN.
未运行完整测试套件并确认全绿，禁止提交。无例外。

# ===== Layer 1: Authority 铁律 + Commitment 宣布 =====

提交前必须先输出：
"我正在执行强制测试验证流程，检查本次所有变更的测试覆盖。"

# ===== Layer 2: 反合理化表格 =====

| 借口 | 现实 |
|------|------|
| "只改了一行注释" | 注释变更也可能破坏文档生成或 lint。跑一次 30 秒。 |
| "相关测试已经覆盖了" | "相关"是你的判断，不是证据。全量跑。 |
| "测试太慢了" | CI 上跑全量更慢。本地先发现问题成本最低。 |

# ===== Layer 3: 顺序门控 + When/Then =====

Step 1: IDENTIFY — 列出本次所有变更文件
Step 2: ANALYZE — 根据用户意图定义应有的测试行为（不看代码，防止实现偏见）
Step 3: RUN — 执行完整测试套件（不只是"相关"测试）
Step 4: READ — 读取完整输出，记录失败数和 exit code
Step 5: VERIFY — 逐文件比对：变更文件是否都有对应测试通过？
        若 NO → 修复 → 回到 Step 3
        若 YES → Step 6
Step 6: CLAIM — 输出测试报告并提交

# When/Then 触发器（可机械判定）
When git diff 显示文件 A 被修改或新增
Then 确认 tests/ 下存在对应 A 的测试文件
     且该测试文件在最新一次运行中状态为 PASS
若任一条件不满足 → STOP，补写测试

# ===== Layer 4: Gate Function =====

"通过"的标准：测试套件 exit code = 0，0 failures。
不允许使用"应该通过了"、"看起来没问题"等措辞。
只有实际命令输出 + exit code 可作为通过的证据。
