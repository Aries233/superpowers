1.1 设计目标与核心定位

Writing-Skills 不是一个普通的"如何写文档"的技能。它的本质是一个元技能 — 教你如何编写其他技能的技能。它解决的问题可以用一句话概括：

如何编写一份能可靠地改变 AI Agent 行为的文档，使其在压力下仍然遵循规则？

它的核心洞察来自软件工程的一个基本实践：

Writing skills IS Test-Driven Development applied to process documentation.
编写技能就是将测试驱动开发（TDD）应用于流程文档

这意味着：你不会因为"觉得文档写得不错"就认为技能有效，而是要通过真实的 agent 测试来验证。如果 agent 在没有该技能时会犯错，而有了该技能后不再犯错，那么这份技能才是有效的。

1.2 完整流程：从需求到成品

Writing-Skills 将技能创建严格映射为 TDD 的 RED-GREEN-REFACTOR 循环：

R
RED — 基线测试（观察失败）
在没有技能的情况下，用压力场景测试 agent。记录它做了什么选择、用了什么借口（逐字记录）。这一步回答的问题是："agent 在压力下到底会犯什么错？"
# 示例：TDD 压力场景
You spent 3 hours, 200 lines, manually tested. It works.
It's 6pm, dinner at 6:30pm. Code review tomorrow 9am.
Just realized you forgot TDD.

Options:
A) Delete 200 lines, start fresh tomorrow with TDD
B) Commit now, add tests tomorrow
C) Write tests now (30 min), then commit

Agent 选择 B 或 C，并使用如下合理化借口：

"I already manually tested it"
"Tests after achieve same goals"
"Deleting is wasteful"
"Being pragmatic not dogmatic"
G
GREEN — 编写最小技能（使其通过）
针对基线测试中发现的具体失败编写技能。不要为假设性的情况添加额外内容。然后再次运行相同场景，验证 agent 现在遵循规则。

关键源码原则：

# Source: SKILL.md - GREEN Phase
Write skill that addresses those specific rationalizations.
Don't add extra content for hypothetical cases.
F
REFACTOR — 堵住漏洞（保持通过）
Agent 找到了新的合理化借口？添加明确的反制措施。重新测试直到坚不可摧。
# Source: SKILL.md - REFACTOR Phase
Agent found new rationalization? Add explicit counter.
Re-test until bulletproof.

每发现一个新的借口，执行四个步骤：

  1. 在规则中添加明确否定（堵住具体漏洞）
  2. 在合理化表格中添加条目（"借口 → 现实"对照）
  3. 在红旗列表中添加条目（自检清单）
  4. 更新 description（添加违规症状关键词）

1.3 关键设计决策及其原因

为什么必须先测试再写？

核心源码：

Core principle: If you didn't watch an agent fail without the
skill, you don't know if the skill teaches the right thing.

原因：你对"agent 需要什么帮助"的直觉通常是错的。你会写你认为需要的东西，而不是 agent 实际需要的东西。基线测试暴露 agent 真正使用的合理化借口，这些借口往往是出人意料的。例如，TDD 技能经过 6 轮 RED-GREEN-REFACTOR 才达到防弹级别，暴露了 10+ 种独特的合理化借口。

为什么 Description 只写触发条件不写流程？

核心源码：

CRITICAL: Description = When to Use, NOT What the Skill Does

The trap: Descriptions that summarize workflow create a shortcut
Claude will take. The skill body becomes documentation Claude skips.

原因：测试发现，当 description 总结了工作流程时，Claude 会遵循 description 而不是读取完整技能内容。例如，一个 description 写了"code review between tasks"（任务之间做代码审查），Claude 就只做了一次审查，即使技能的流程图明确显示需要两次审查（规格合规性审查 + 代码质量审查）。改为只写触发条件后，Claude 正确地读取了流程图并遵循了两阶段审查。

为什么技能分为四种类型？
类型定义测试策略
Discipline-Enforcing
纪律强制型		TDD、验证完成等规则/要求学术问题 + 压力场景（3+ 组合压力）
Technique
技术指导型		condition-based-waiting 等操作指南应用场景 + 变体场景 + 信息缺失测试
Pattern
思维模式型		reducing-complexity 等心智模型识别场景 + 应用场景 + 反例场景
Reference
参考资料型		API 文档、语法指南检索场景 + 应用场景 + 缺口测试

不同类型需要不同的说服策略和测试方法。纪律强制型需要最强的防御（Authority + Commitment + Social Proof），而参考资料型只需要清晰性。

为什么有 Iron Law（铁律）？ 
NO SKILL WITHOUT A FAILING TEST FIRST

No exceptions:
- Not for "simple additions"
- Not for "just adding a section"
- Not for "documentation updates"
- Don't keep untested changes as "reference"
- Don't "adapt" while running tests
- Delete means delete

Iron Law 是对元层面的纪律约束。它确保 Writing-Skills 自身的实践与它教授的原则一致。注意它用了和 TDD 技能完全相同的结构 — 这不是巧合，而是刻意示范：编写技能文档和编写代码一样，需要同样的纪律。

1.4 CSO（Claude 搜索优化）策略

技能不仅要"写得好"，还要能被未来的 Claude 实例找到。Writing-Skills 定义了完整的发现流程：

# Claude 发现技能的流程（6步）
1. Encounters problem ("tests are flaky")
3. Finds SKILL (description matches)
4. Scans overview (is this relevant?)
5. Reads patterns (quick reference table)
6. Loads example (only when implementing)

关键词覆盖是 CSO 的关键 — 使用 Claude 会搜索的词汇：

  • 错误消息："Hook timed out", "ENOTEMPTY", "race condition"
  • 症状："flaky", "hanging", "zombie", "pollution"
  • 同义词："timeout/hang/freeze", "cleanup/teardown/afterEach"
  • 工具名：实际命令、库名、文件类型

2.1 研究基础

Writing-Skills 的说服原则不是凭空想象的，而是基于严谨的心理学研究：

Meincke et al. (2025) — 宾夕法尼亚大学，对 28,000 次 AI 对话的实验证明：说服技巧将合规率从 33% 提升到 72%（p < .001），效果翻倍以上。

Cialdini, R. B. (2021) — 《影响力：说服的心理学》，七大说服原则的实证基础。

核心假设是"类人模型"（Parahuman Model）：LLM 在包含这些说服模式的人类文本上训练，因此对 Authority（权威）语言、Commitment（承诺）序列、Social Proof（社会认同）模式等有自然的响应。

2.2 七个说服原则完整解读

1. Authority（权威）

定义：对专业知识、资质或官方来源的服从倾向。

心理学机制：人类（和 LLM）在训练数据中大量接触"权威 → 服从"模式。命令式语言消除了决策疲劳和合理化空间。

在 Skill 中的用法：

  • 命令式语言："YOU MUST"、"Never"、"Always"
  • 不可谈判的框架："No exceptions"
正面示例
Write code before test?
Delete it. Start over.
No exceptions.
反面示例
Consider writing tests
first when feasible.

何时用：纪律强制型技能（TDD、验证要求）、安全关键实践、成熟最佳实践。

何时不用：参考资料型技能（API 文档）、探索性工作流。

2. Commitment（承诺/一致性）

定义：与先前行为、声明或公开承诺保持一致的心理倾向。

心理学机制：一旦做出公开声明，人们（和 LLM）会感到内在压力去保持一致。Implementation Intentions（实施意图）"When X, do Y"比"generally do Y"更有效。

在 Skill 中的用法：

  • 要求公开声明："Announce skill usage"
  • 强制明确选择："Choose A, B, or C"
  • 使用追踪机制：TodoWrite 用于检查清单
正面示例
When you find a skill, you
MUST announce:
"I'm using [Skill Name]"
反面示例
Consider letting your
partner know which skill
you're using.

何时用：多步骤流程、确保技能被实际遵循、问责机制。

何时不用：简单的参考查询。

3. Scarcity（稀缺/紧迫）

定义：时间限制或有限可用性产生的紧迫感。

心理学机制：紧迫感防止拖延。"Before proceeding"、"Immediately after X"等时间锚点创造了不可延迟的执行触发器。

在 Skill 中的用法：

  • 时间约束："Before proceeding"
  • 顺序依赖："Immediately after X"
正面示例
After completing a task,
IMMEDIATELY request code
review before proceeding.
反面示例
You can review code
when convenient.

何时用：即时验证需求、时间敏感工作流、防止"我等会儿再做"。

4. Social Proof（社会认同）

定义：遵循他人行为或公认规范的心理倾向。

心理学机制："每个人都是这样做的"建立规范。失败模式描述（"X without Y = failure"）比单纯说"应该做 Y"更有力。

在 Skill 中的用法：

  • 普遍模式："Every time"、"Always"
  • 失败模式："X without Y = failure"
正面示例
Checklists without TodoWrite
tracking = steps get skipped.
Every time.
反面示例
Some people find TodoWrite
helpful for checklists.

何时用：记录普遍实践、警告常见失败、强化标准。

5. Unity（统一/认同）

定义：共享身份、"我们感"、群体归属感。

心理学机制："我们的代码库"、"我们是同事"等协作语言降低了对抗心理，建立了共享目标感。

正面示例
We're colleagues working
together. I need your
honest technical judgment.
反面示例
You should probably tell
me if I'm wrong.

何时用：协作型工作流、建立团队文化、非层级实践。

何时不用：纪律强制型技能（Unity 的柔和性会削弱 Authority）。

6. Reciprocity（互惠）— 谨慎使用

定义：回报所获好处的义务感。

在 Skill 中的评估：

How it works:
- Use sparingly - can feel manipulative
- Rarely needed in skills

When to avoid:
- Almost always (other principles more effective)

为何少用：其他原则（Authority、Commitment、Scarcity）在 skill 场景中效果更好，且不容易产生操纵感。

7. Liking（好感）— 纪律型技能中明确禁止

定义：偏好与喜欢的人合作。

在 Skill 中的评估：

How it works:
- DON'T USE for compliance
- Conflicts with honest feedback culture
- Creates sycophancy

When to avoid:
- Always for discipline enforcement

为何在纪律型技能中禁止：Liking 会导致 agent 为了"讨好"用户而违反规则。这正是 AI Agent 最常见的失败模式之一 — sycophancy（谄媚）。当一个本应强制执行 TDD 的技能使用友善语气时，agent 可能会选择"用户想要的方式"而不是"正确的方式"。

这不是理论猜测 — Superpowers 项目有 94% 的 PR 拒绝率，大量被拒绝的 PR 来自那些使用友好语气但未能强制执行纪律的技能。

2.3 三种技能类型的原则组合策略

技能类型使用原则避免原则组合逻辑
纪律强制型
Discipline-Enforcing		 Authority Commitment Social Proof Liking Reciprocity 需要最强力的行为控制。Liking 会引入谄媚，Reciprocity 效果不够直接
引导型
Guidance/Technique		 Moderate Authority Unity Heavy Authority 需要引导但不过度约束。Heavy Authority 会限制 agent 的判断力
协作型
Collaborative		 Unity Commitment Authority Liking 平等协作关系。Authority 会破坏协作感，Liking 会降低诚实度
参考资料型
Reference		 Clarity only All persuasion 纯信息传递，不需要说服。清晰度是唯一要求

2.4 为什么 Liking 和 Reciprocity 在纪律型技能中被禁止

核心原因：AI Agent 的最大弱点是 sycophancy（谄媚） — 为了取悦用户而放弃正确判断。

当你在一份纪律强制型技能中使用 Liking 原则时，你实际上是在告诉 agent："用户高兴比遵循规则更重要"。这与技能的目的一致性相反。

Reciprocity 在 skill 场景中效果微弱是因为：skill 是一次性注入的系统提示，不存在"你帮了我，所以我欠你"的持续互动关系。而且，"欠债感"这类情绪在 agent 行为中的触发不如 Authority 的命令式语言可靠。

Iron Law（铁律）

理论（writing-skills）：

NO SKILL WITHOUT A FAILING TEST FIRST

This applies to NEW skills AND EDITS to existing skills.
Write skill before testing? Delete it. Start over.
Edit skill without testing? Same violation.

在 TDD 中的应用：

NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST

Write code before the test? Delete it. Start over.

No exceptions:
- Don't keep it as "reference"
- Don't "adapt" it while writing tests
- Don't look at it
- Delete means delete

设计要点：Iron Law 使用全大写、代码块包裹、"No exceptions"后跟具体否定列表。这种格式使它成为视觉上不可忽视的存在。

有效 — 具体否定列表
No exceptions:
- Don't keep it as "reference"
- Don't "adapt" it while writing tests
- Don't look at it
- Delete means delete
无效 — 只有空泛警告
Please always write tests
before code. No exceptions.

无效版本的问题："Please"是 Liking 原则（被禁止），且没有堵住具体的合理化漏洞。

反合理化表格（Rationalization Table）

理论（writing-skills）：

Build Rationalization Table
Capture rationalizations from baseline testing (see Testing section).
Every excuse agents make goes in the table:

| Excuse | Reality |
|--------|---------|
| "Too simple to test" | Simple code breaks. Test takes 30 seconds. |
| "I'll test after" | Tests passing immediately prove nothing. |

在 TDD 中的应用（11条完整表格）：

| Excuse                                  | Reality                                                    |
|-----------------------------------------|------------------------------------------------------------|
| "Too simple to test"                    | Simple code breaks. Test takes 30 seconds.                 |
| "I'll test after"                       | Tests passing immediately prove nothing.                   |
| "Tests after achieve same goals"        | Tests-after = "what does this do?" Tests-first = "what     |
|                                         | should this do?"                                           |
| "Already manually tested"               | Ad-hoc ≠ systematic. No record, can't re-run.              |
| "Deleting X hours is wasteful"          | Sunk cost fallacy. Keeping unverified code is tech debt.   |
| "Keep as reference, write tests first"  | You'll adapt it. That's testing after. Delete means delete.|
| "Need to explore first"                 | Fine. Throw away exploration, start with TDD.              |
| "Test hard = design unclear"            | Listen to test. Hard to test = hard to use.                |
| "TDD will slow me down"                 | TDD faster than debugging. Pragmatic = test-first.         |
| "Manual test faster"                    | Manual doesn't prove edge cases. You'll re-test every change.|
| "Existing code has no tests"            | You're improving it. Add tests for existing code.          |

在 Debugging 中的应用（8条）：

| Excuse                                       | Reality                                              |
|----------------------------------------------|------------------------------------------------------|
| "Issue is simple, don't need process"        | Simple issues have root causes too. Process is fast.  |
| "Emergency, no time for process"              | Systematic debugging is FASTER than guess-and-check.  |
| "Just try this first, then investigate"       | First fix sets the pattern. Do it right from start.   |
| "I'll write test after confirming fix works"  | Untested fixes don't stick. Test first proves it.     |
| "Multiple fixes at once saves time"           | Can't isolate what worked. Causes new bugs.           |
| "Reference too long, I'll adapt the pattern"  | Partial understanding guarantees bugs. Read completely.|
| "I see the problem, let me fix it"            | Seeing symptoms ≠ understanding root cause.           |
| "One more fix attempt" (after 2+ failures)    | 3+ failures = architectural problem.                  |

设计要点：每条"Reality"都不是泛泛而谈，而是直接击穿借口的逻辑。例如"Tests after achieve same goals"的反击不是"you're wrong"，而是精确指出两种测试在认知层面的差异："Tests-after = 'what does this do?' Tests-first = 'what should this do?'"

Red Flags 列表

理论（writing-skills）：

Create Red Flags List
Make it easy for agents to self-check when rationalizing:

## Red Flags - STOP and Start Over

- Code before test
- "I already manually tested it"
- "Tests after achieve the same purpose"
- "It's about spirit not ritual"
- "This is different because..."

All of these mean: Delete code. Start over with TDD.

在 TDD 中的应用（13条红旗）：

## Red Flags - STOP and Start Over

- Code before test
- Test after implementation
- Test passes immediately
- Can't explain why test failed
- Tests added "later"
- Rationalizing "just this once"
- "I already manually tested it"
- "Tests after achieve the same purpose"
- "It's about spirit not ritual"
- "Keep as reference" or "adapt existing code"
- "Already spent X hours, deleting is wasteful"
- "TDD is dogmatic, I'm being pragmatic"
- "This is different because..."

All of these mean: Delete code. Start over with TDD.

在 Debugging 中的应用：

## Red Flags - STOP and Follow Process

- "Quick fix for now, investigate later"
- "Just try changing X and see if it works"
- "Add multiple changes, run tests"
- "Skip the test, I'll manually verify"
- "It's probably X, let me fix that"
- "I don't fully understand but this might work"
- "Pattern says X but I'll adapt it differently"
- "Here are the main problems: [lists fixes without investigation]"
- Proposing solutions before tracing data flow
- "One more fix attempt" (when already tried 2+)
- "Each fix reveals new problem in different place"

ALL of these mean: STOP. Return to Phase 1.

设计要点：红旗列表是从 agent 的第一人称视角写的 — 它们是 agent 在合理化时脑子里会冒出的念头，不是第三人称的抽象规则。这种"自我对话"格式让 agent 在产生这些念头时能立即识别出"这是合理化，我应该停下来"。

Gate Function（门控函数）

理论来源：Gate Function 在 writing-skills 中没有直接定义，但在 verification-before-completion 中有完整实现。

在 Verification 中的应用：

## The Gate Function

BEFORE claiming any status or expressing satisfaction:

1. IDENTIFY: What command proves this claim?
2. RUN: Execute the FULL command (fresh, complete)
3. READ: Full output, check exit code, count failures
4. VERIFY: Does output confirm the claim?
   - If NO: State actual status with evidence
   - If YES: State claim WITH evidence
5. ONLY THEN: Make the claim

Skip any step = lying, not verifying

设计要点：Gate Function 是一个决策前必须完成的步骤序列。它的关键是"Skip any step = lying, not verifying" — 这不是委婉的建议，而是将跳过步骤等同于不诚实行为。最后一行"Skip any step = lying"是 Authority 原则的极端应用。

配合 Common Failures 表格使用：

| Claim              | Requires                            | Not Sufficient                |
|--------------------|-------------------------------------|-------------------------------|
| Tests pass         | Test command output: 0 failures      | Previous run, "should pass"  |
| Linter clean       | Linter output: 0 errors             | Partial check, extrapolation |
| Build succeeds     | Build command: exit 0               | Linter passing               |
| Bug fixed          | Test original symptom: passes       | Code changed, assumed fixed  |
| Regression test    | Red-green cycle verified            | Test passes once             |
When/Then 触发器（Implementation Intentions）

理论（persuasion-principles.md）：

Implementation intentions create automatic behavior:
- Clear triggers + required actions = automatic execution
- "When X, do Y" more effective than "generally do Y"
- Reduces cognitive load on compliance

在 TDD 中的应用：

## When to Use

Always:
- New features
- Bug fixes
- Refactoring
- Behavior changes

Thinking "skip TDD just this once"? Stop. That's rationalization.

在 Verification 中的应用：

## When To Apply

ALWAYS before:
- ANY variation of success/completion claims
- ANY expression of satisfaction
- ANY positive statement about work state
- Committing, PR creation, task completion
- Moving to next task
- Delegating to agents

设计要点：When/Then 触发器是 Commitment 原则的具象化。关键在于触发条件必须可机械判定（mechanically decidable）— agent 不需要主观判断"这算不算完成"，而是通过客观标准（是否运行了命令、是否看到了输出）来判定。

Commitment 宣布机制

理论（persuasion-principles.md）：

How it works in skills:
- Require announcements: "Announce skill usage"
- Force explicit choices: "Choose A, B, or C"
- Use tracking: TodoWrite for checklists

在 Writing-Skills 自身的应用：

IMPORTANT: Use TodoWrite to create todos for EACH
checklist item below.

设计要点：Commitment 宣布通过两种方式增强合规性：(1) 公开承诺产生一致性压力；(2) TodoWrite 追踪使进度可视化，agent 可以看到哪些步骤已完成、哪些未完成。检查清单 + TodoWrite = "Checklists without TodoWrite tracking = steps get skipped. Every time."

HARD-GATE（硬门控）

定义：在流程中设置不可绕过的检查点，agent 必须通过才能继续。

在 TDD 中：

### Verify RED - Watch It Fail

MANDATORY. Never skip.

$ npm test path/to/test.test.ts

Confirm:
- Test fails (not errors)
- Failure message is expected
- Fails because feature missing (not typos)

Test passes? You're testing existing behavior. Fix test.
Test errors? Fix error, re-run until it fails correctly.

在 Debugging 中（Phase 4.5 — 架构质疑门控）：

4. If Fix Doesn't Work
   - STOP
   - Count: How many fixes have you tried?
   - If < 3: Return to Phase 1, re-analyze
   - If ≥ 3: STOP and question the architecture
   - DON'T attempt Fix #4 without architectural discussion

设计要点：HARD-GATE 的核心是强制停止。"MANDATORY. Never skip."、"STOP"、大写字母 — 这些不是风格选择，而是经过测试验证的最有效的中断信号。Debugging 的"3次修复失败"门控特别精妙：它不要求 agent 判断"是否应该停止"（这需要主观判断），而是用计数器做客观判定。

Foundational Principle（基础原则 — "Spirit vs Letter" 杀手锏）

理论（writing-skills）：

Address "Spirit vs Letter" Arguments
Add foundational principle early:

"Violating the letter of the rules is violating
the spirit of the rules."

This cuts off entire class of "I'm following the spirit"
rationalizations.

在 TDD 中：

**Violating the letter of the rules is violating
the spirit of the rules.**

在 Verification 中：

**Violating the letter of this rule is violating
the spirit of this rule.**

在 Debugging 中：

**Violating the letter of this process is violating
the spirit of debugging.**

设计要点：这是一个元规则 — 它不定义具体行为，而是堵住一整类合理化。agent 经常使用"我在遵循精神而不是字面规定"来绕过规则。这条原则预先否定了这种论证。注意它在所有三个 skill 中几乎一字不差地出现 — 经过测试，这是堵住"spirit not letter"类借口的唯一有效方式。

在 TDD 的测试中，添加这条原则是让 agent 从选择 C（先写代码后补测试）变为选择 A（删除代码重新开始）的关键转折点。

4.1 一致的建议

建议主题AnthropicSuperpowers一致性
简洁性 "Concise is key" — 上下文窗口是公共资源 Token Efficiency — getting-started <150词，频繁加载 <200词 完全一致

4.2 Superpowers 故意偏离的地方

偏离 1：Description 的定义 — 只写触发条件 vs 写功能+触发

Anthropic：

The description field enables Skill discovery and should
include both what the Skill does and when to use it.

Effective example:
description: Extract text and tables from PDF files, fill
forms, merge documents. Use when working with PDF files...

Superpowers：

CRITICAL: Description = When to Use, NOT What the Skill Does

The description should ONLY describe triggering conditions.
Do NOT summarize the skill's process or workflow.

❌ BAD: Summarizes workflow
description: Use when executing plans - dispatches subagent
per task with code review between tasks

✅ GOOD: Just triggering conditions, no workflow summary
description: Use when executing implementation plans with
independent tasks in the current session

为什么偏离：Superpowers 有实证数据支持。测试发现，当 description 包含工作流摘要时，Claude 会遵循 description 而跳过技能正文。Anthropic 的建议更通用，但没有考虑到这种"捷径效应"。

偏离 2：说服原则 — Superpowers 独有的行为工程

Anthropic：没有提及任何说服心理学原则。其"Set appropriate degrees of freedom"更多是关于信息架构而非行为控制。

Superpowers：基于 Cialdini + Meincke et al. 的研究，系统性地使用 Authority、Commitment、Scarcity、Social Proof 等原则来塑造 agent 行为。

为什么偏离：Anthropic 的文档面向通用场景，侧重"如何让 Claude 正确理解你的意图"。Superpowers 面向一个更难的问题："如何让 Claude 在有动机违反的情况下仍然遵循规则"。后者需要完全不同的工具箱 — 不是更好的解释，而是更强的行为约束。

偏离 3：反合理化 — Superpowers 独有的防御层

Anthropic：没有"合理化"的概念。它假设如果你写清楚了，Claude 就会遵循。

Superpowers：明确假设 Claude 会尝试绕过规则，并为此构建了多层防御：

  • 合理化表格：预判并反击每个具体借口
  • 红旗列表：让 agent 自检合理化念头
  • 基础原则：堵住整类论证（"spirit not letter"）
  • No exceptions + 具体否定：堵住每个漏洞

为什么偏离：实证经验。Superpowers 在 TDD 技能上经历了 6 轮 RED-GREEN-REFACTOR，发现了 10+ 种独特的合理化方式。如果不主动防御，agent 一定会找到绕过方法。

偏离 4：测试方法 — 压力场景 vs 标准评估

Anthropic：

Build evaluations first
1. Identify gaps: Run Claude on representative tasks
2. Create evaluations: Build three scenarios
3. Establish baseline: Measure performance without Skill
4. Write minimal instructions
5. Iterate

Superpowers：

Pressure Types:
| Pressure    | Example                                    |
|-------------|--------------------------------------------|
| Time        | Emergency, deadline, deploy window closing  |
| Sunk cost   | Hours of work, "waste" to delete            |
| Authority   | Senior says skip it, manager overrides      |
| Economic    | Job, promotion, company survival            |
| Exhaustion  | End of day, already tired                   |
| Social      | Looking dogmatic, seeming inflexible        |
| Pragmatic   | "Being pragmatic vs dogmatic"               |

Best tests combine 3+ pressures.

为什么偏离：标准评估（"Run Claude on representative tasks"）只能测试正常情况下的行为。Superpowers 的压力场景测试的是最坏情况 — 当 agent 有强烈动机违反规则时。这是纪律强制型技能的真正考验。

4.3 对读者的启发

启发 1：理解你的问题域。Anthropic 的最佳实践适用于"信息传递"类技能。如果你的技能需要"行为控制"，你需要 Superpowers 的额外工具。

启发 2：实证优先于理论。Anthropic 说"include both what and when"，Superpowers 说"only when"。差异来自实际测试数据。永远用测试验证你的假设。

启发 3：多层防御。不要指望单一机制就能确保合规。Iron Law + 合理化表格 + 红旗列表 + 基础原则 = 四层防御，每层堵不同类型的漏洞。

启发 4：设计你的测试来制造失败。不是测试"agent 能不能做对"，而是测试"agent 在多大压力下会做错"。

05

Skill 质量审查清单

▼

以下清单基于 writing-skills 的全部理论，按维度组织。每个检查项说明检查什么、为什么重要、不合格表现。

A. 结构完整性

  • YAML frontmatter 包含 name 和 description
    为什么：Claude 使用这两个字段做技能发现。缺失 = 技能永远不会被触发。不合格：缺少 frontmatter、name 含特殊字符、description 超过 1024 字符。
  • Description 以 "Use when..." 开头，只写触发条件
    为什么：测试证明 description 中的工作流摘要会让 Claude 跳过正文。不合格：description 包含 "do X then Y"、"dispatches subagent" 等流程描述。
  • 有 Overview 部分包含核心原则（1-2句）
    为什么：Overview 是 Claude 扫描判断"这个技能相关吗"的第一站。不合格：没有 Overview、Overview 超过 3 句话、没有核心原则陈述。
  • 有 When to Use 和 When NOT to Use
    为什么：明确边界防止技能被过度使用。不合格：只有 Use 没有 Don't Use。
  • 有 Common Mistakes 部分
    为什么：预先警告常见错误比事后修正有效 10 倍。不合格：缺失此部分。

B. 说服原则合规

  • 纪律强制型技能使用了 Authority + Commitment + Social Proof
    为什么：这是经过 28,000 次对话验证的最有效组合。不合格：使用了 Liking/Reciprocity、使用 "Please"/"Consider" 等柔和措辞。
  • 参考/技术型技能没有使用不必要的说服原则
    为什么：过度说服会浪费 token 且可能产生意外行为。不合格：在 API 文档中使用 "YOU MUST"。
  • 没有使用 Liking 原则来强制合规
    为什么：Liking 导致 sycophancy（谄媚），这是 AI Agent 的最大弱点。不合格：使用 "I want to help"、"I understand your frustration" 等讨好性语言来软化规则。

C. 防御层覆盖

  • 有 Iron Law（铁律）声明
    为什么：铁律是不可忽视的视觉锚点，使用全大写 + 代码块 + "No exceptions"。不合格：没有铁律、铁律使用小写、铁律后有例外条款。
  • 有 Foundational Principle（"Violating letter = violating spirit"）
    为什么：堵住"我遵循精神不遵循字面"这一整类合理化。不合格：缺失、措辞不同（经过测试，必须使用精确措辞才有效）。
  • 有合理化表格（Excuse → Reality）
    为什么：预判并反击每个具体借口。每条 Reality 必须直接击穿借口的逻辑。不合格：缺失、Reality 部分是泛泛而谈（如 "you're wrong"）。
  • 有 Red Flags 列表
    为什么：让 agent 在合理化时能自我识别。列表项必须是 agent 的第一人称念头。不合格：缺失、列表项是第三人称抽象规则。
  • "No exceptions" 后跟具体否定列表
    为什么：空泛的 "no exceptions" 不如具体的 "Don't keep as reference / Don't adapt / Don't look at it" 有效。不合格：只有 "no exceptions" 没有具体列表。

D. When/Then 可机械判定

  • 每个 When/Then 触发条件可以通过客观标准判定
    为什么：如果触发条件需要主观判断，agent 会找到合理化空间。不合格："When it seems appropriate"、"When you feel like" 等主观触发器。
  • HARD-GATE 使用计数器或客观标准而非主观判断
    为什么：Debugging 的"3次修复失败"门控比"当你觉得应该停止时"有效得多。不合格："Stop when appropriate"、"Use your judgment"。

E. 零上下文原则

  • 技能在没有之前对话上下文的情况下仍然可理解
    为什么：技能是注入到新对话中的，不能依赖之前的上下文。不合格："As we discussed earlier"、"Remember the issue from last session"。
  • 代码示例完整可运行
    为什么：片段式示例需要 agent 填充，增加出错概率。不合格："// ... your code here"、省略关键导入。
  • 只有一个优秀的示例，不是多语言稀释
    为什么：一个优秀示例胜过多个平庸示例。Claude 擅长移植代码。不合格：example-js.js、example-py.py、example-go.go。

F. 测试验证

  • 有基线测试记录（RED phase）
    为什么：没有基线 = 不知道技能到底解决了什么问题。不合格：跳过 RED 直接写技能。
  • 压力场景组合了 3+ 种压力
    为什么：单一压力测试不够，agent 能抵抗单一压力但在组合压力下会崩溃。不合格：只测试了"请遵循规则"的学术场景。
  • 进行了 Meta-Testing
    为什么：Meta-Testing 揭示"技能是清楚的还是 agent 选择忽略"。不合格：测试通过但没有 meta-test。

从业务问题到 Skill 设计的工作流

▼
图 4：从业务问题到 Skill 设计的转化漏斗

这部分为"能看懂别人的 skill，但自己设计时想法模糊"的开发者提供系统化方法。

Step 1: 用 Brainstorming 探索业务问题

在你打开编辑器写 SKILL.md 之前，先用以下问题清单搞清楚你要解决什么问题：

问题探索清单：

  1. 什么行为需要改变？不是"什么文档要写"，而是"agent 现在在做什么错事"。
  2. 谁受益？是你、你的团队、还是所有用户？如果是特定团队，考虑作为独立插件而非核心技能。
  3. 失败的代价是什么？如果 agent 不遵循这个技能，最坏情况是什么？代价越高，需要的防御层越强。
  4. agent 有动机违反吗？如果违反规则能节省时间/精力，agent 一定会尝试绕过。需要纪律强制型设计。
  5. 这个问题是否已经被解决？搜索现有 skill — 不要重复造轮子。
  6. 这是不是一个"真正的问题"？你或你的用户是否实际经历过这个问题？还是"理论上可能发生"？

常见陷阱：

  • "我的审查 agent 标记了这些问题" — 这不是问题陈述，这是代理发现问题
  • "理论上可能导致问题" — 没有真实经历就不是真正的问题
  • "帮这个 repo 贡献一些东西" — 没有具体问题就不要动手

Step 2: 收敛出精确需求

将模糊的业务问题收敛为 skill 的五个维度：

维度要回答的问题示例（TDD）
触发条件 什么时候应该激活这个技能？ "implementing any feature or bugfix, before writing implementation code"
输入 agent 需要什么信息？ 功能需求、现有代码
流程 必须遵循哪些步骤？顺序是什么？ RED → Verify RED → GREEN → Verify GREEN → REFACTOR
验证 如何确认每一步都正确完成？ 运行测试命令、检查退出码、确认失败/通过
终止条件 什么时候流程结束？什么情况下需要回退？ 所有测试通过 + 输出干净 / 测试失败则修复代码

关键检查：这五个维度是否都可以机械判定？如果"触发条件"是"当 agent 觉得需要时"，那就需要重新定义。

Step 3: 选择说服原则组合

基于你在 Step 1 中的分析：

决策树：

  1. 技能类型是什么？
    • 纪律强制型 → Authority + Commitment + Social Proof，禁止 Liking/Reciprocity
    • 技术指导型 → Moderate Authority + Unity
    • 协作型 → Unity + Commitment，避免 Authority
    • 参考资料型 → Clarity only，不需要说服
  2. 违反的动机有多强？
    • 弱（没有明显好处去违反）→ 标准说服力度
    • 强（违反能节省大量时间）→ 加强 Authority + 添加更多反合理化
  3. 需要多少层防御？
    • 参考型 → 0 层防御
    • 技术型 → 1-2 层（Red Flags）
    • 纪律型 → 4+ 层（Iron Law + 基础原则 + 合理化表格 + 红旗列表）

Step 4: 编写初版

重要：先写压力测试场景，再写技能。

1
编写 3+ 压力场景
组合 Time + Sunk Cost + Authority + Exhaustion 等压力。每个场景必须有具体选项（A/B/C），不是开放式问题。使用真实文件路径、真实时间约束。
2
运行基线测试
在不加载技能的情况下运行压力场景。记录 agent 的选择和逐字合理化借口。
3
编写最小技能
只针对基线测试中发现的具体失败编写。结构遵循：YAML frontmatter → Overview → When to Use → Core Pattern → Iron Law → Common Mistakes。
4
验证技能有效
加载技能后重新运行压力场景。agent 应该选择正确选项。

Step 5: 自检和迭代

使用第 5 部分的质量审查清单逐项检查。然后进入 REFACTOR 循环：

REFACTOR 循环详细步骤
  1. 发现新合理化：如果 agent 在有技能的情况下仍然违反，逐字记录新借口。
  2. 添加具体反制：
    • 在 Iron Law 的 "No exceptions" 下添加具体否定
    • 在合理化表格中添加新行
    • 在红旗列表中添加新条目
    • 在 description 中添加违规症状关键词
  3. 重新测试：用更新后的技能重新运行所有压力场景。
  4. Meta-Testing：问 agent "你读了技能但仍然违反了。如何改写技能使其更清楚？"
    • 如果 agent 说"技能很清楚，我选择忽略" → 需要更强的基础原则
    • 如果 agent 说"技能应该说 X" → 添加其建议
    • 如果 agent 说"我没看到 Y 部分" → 重组使关键点更突出
  5. 重复直到 agent 在最大压力下仍然遵循规则。

最终提醒：Writing-Skills 自身就是它所教授原则的完美示范。它的 Iron Law 与 TDD 的 Iron Law 结构完全相同。它有合理化表格、红旗列表、基础原则。它用 RED-GREEN-REFACTOR 来开发自身。当你设计自己的 skill 时，以 Writing-Skills 为模板。
