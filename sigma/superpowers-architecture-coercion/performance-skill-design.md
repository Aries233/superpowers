# URL 页面性能排查 Skill — 设计方案

> 本文档是 brainstorming 阶段的产出，包含需求、架构、分析逻辑和报告格式。
> 下一步：迁移到项目中，结合实际代码实现。

## 1. 触发条件

用户提供一个线上报表 URL，要求分析其性能。

## 2. 执行环境

- Claude Code（在源码仓库中执行）
- Playwright（打开浏览器访问目标页面）
- 已有 skill：获取组件元数据和页面元数据

## 3. 信息收集（四层）

### Layer 0: 常规 Web 性能基线
- 采集方式：Playwright 性能 API / Lighthouse
- 指标：TTFB, FCP, LCP, CLS, 页面加载总时间
- 资源：JS/CSS/图片/字节数量与体积
- 网络：请求总数、瀑布图、阻塞情况
- DOM：节点数量、嵌套深度

### Layer 1: 业务接口性能
- 采集方式：Playwright 拦截网络请求 + React Fiber 获取组件实例
- 指标：
  - 元数据接口响应时间
  - 每个组件的 chart/data 接口：URL、响应时间、返回数据量
  - 重复请求检测
  - 串行阻塞检测

### Layer 2: 数据层性能
- 采集方式：读取组件关联的 SQL 查询 + 数据集配置
- 指标：
  - 每个组件生成的 SQL
  - SQL 涉及的表（宽表/明细表/维度表）
  - WHERE/GROUP BY/JOIN 分析
  - 查询耗时

### Layer 3: 渲染性能
- 采集方式：Playwright Performance API
- 指标：
  - 每个组件的渲染耗时
  - 总渲染耗时
  - 组件间是否串行阻塞

## 4. 性能标准（默认阈值，可配置）

| 指标 | 良好 | 待改进 | 差 |
|------|------|--------|-----|
| LCP | < 2.5s | 2.5-4s | > 4s |
| FCP | < 1.8s | 1.8-3s | > 3s |
| TTFB | < 800ms | 800-1800ms | > 1800ms |
| 元数据接口 | < 500ms | 500ms-1s | > 1s |
| 组件数据接口 | < 1s | 1-3s | > 3s |
| SQL 查询 | < 200ms | 200ms-1s | > 1s |
| 单组件渲染 | < 200ms | 200-500ms | > 500ms |

## 5. 分析逻辑

### 5.1 总体判断
1. 所有指标与标准比对，标记"良好/待改进/差"
2. 识别瓶颈：哪个 layer 的指标最差？

### 5.2 分层分析

**Layer 0 不达标时：**
- 资源体积过大 → 具体指出哪个 JS/CSS/图片文件，建议压缩/拆分
- HTTP 版本 → 检测是否使用 HTTP/1.1，建议升级 HTTP/2
- DOM 过大 → 统计节点数，建议减少

**Layer 1 不达标时：**
- 慢接口 → 关联到具体组件名称，分析其 SQL（进入 Layer 2）
- 重复请求 → 列出具体哪几个组件请求了相同接口
- 返回数据量大 → 标记具体接口和返回大小

**Layer 2 不达标时：**
- WHERE 无索引 → 指出具体是哪个字段，建议添加索引
- 大表全表扫描 → 分析是宽表还是明细表，建议表结构调整
- JOIN 复杂 → 指出关联了哪几张表
- 数据集配置问题 → 指出具体数据集名称和建议的配置调整

**Layer 3 不达标时：**
- 渲染慢 → 追溯是否由 Layer 1 的数据接口拖慢
- 组件过多 → 统计组件数，建议合并相似组件（指出具体哪几个）

## 6. 报告输出格式

### 第一部分：给搭建者的优化建议

格式要求：每个建议必须包含具体的组件/数据集/字段名称。

```
## 报表性能分析报告

### 总体评分：[良好/待改进/差] — 主要瓶颈在 [Layer X]

---

### 🔧 你可以立即优化的项目

#### 1. 合并数据组件
- 问题：组件「销售趋势图」和组件「月度对比图」使用了相同的数据集「销售明细」
- 建议：将这两个组件合并为一个组件，使用数据联动
- 预期收益：减少 1 次 chart/data 请求，预计节省 [X]ms

#### 2. 调整数据集配置
- 问题：组件「区域分布图」使用的数据集「订单宽表」包含 [N] 个字段，但只用了 3 个
- 建议：将数据集从「订单宽表」切换为「订单明细表」，只关联需要的维度字段
- 预期收益：减少数据扫描量，查询预计从 [X]ms 降至 [Y]ms

#### 3. 减少组件数量
- 问题：页面共 [N] 个组件，超过了 [阈值] 的推荐值
- 建议：移除/合并以下组件：[组件A名称] 和 [组件B名称] 展示相同维度的数据
- 预期收益：减少 [X] 个网络请求，降低页面复杂度

---

### 📋 建议提工单的平台优化项

#### 1. 重复接口请求
- 问题：[接口URL] 在页面加载时被调用了 [N] 次
- 建议：平台侧应实现请求去重/缓存

#### 2. 慢 SQL 需要索引
- 问题：[具体SQL] 查询耗时 [X]ms，WHERE 条件字段「[字段名]」无索引
- 建议：联系 DBA 为 [表名].[字段名] 添加索引
```

### 第二部分：给开发人员的优化方向

```
### 🔨 开发侧优化方向

#### 1. 接口合并机会
- 位置：[源码模块路径]
- 问题：[组件A] 和 [组件B] 的 chart/data 请求参数相似，可合并
- 建议：在 [模块] 中实现批量查询接口

#### 2. 重复请求
- 位置：[源码模块路径]
- 问题：[接口URL] 在 [组件A] 和 [组件B] 中被分别调用
- 建议：提升数据请求到父组件层级，通过 props 传递

#### 3. 渲染优化
- 位置：[源码模块路径]
- 问题：[N] 个组件串行渲染，总耗时 [X]ms
- 建议：实现组件懒加载或虚拟滚动
```

## 7. Skill 的设计模式应用

基于 superpowers 的设计原则，本 skill 应使用以下模式：

| 模式 | 在本 skill 中的应用 |
|------|-------------------|
| **Iron Law** | "NO PERFORMANCE REPORT WITHOUT ACTUAL MEASUREMENT. 不采集不分析。" |
| **Gate Function** | 每个分析步骤必须基于采集到的真实数据，不允许猜测 |
| **Sequential Gating** | Layer 0 → Layer 1 → Layer 2 → Layer 3 顺序执行，不跳层 |
| **Anti-Rationalization** | "页面看起来挺快的" → "用数据说话，不用感觉判断" |
| **Zero-Context Handoff** | 报告中的每个建议必须自包含：具体组件名 + 具体问题 + 具体建议 |
| **When/Then Trigger** | When 组件接口 > 3s → Then 追溯 SQL 分析 + 数据集配置分析 |
| **Commitment** | 开始时宣布："我正在使用性能排查 skill 分析 [URL]" |
