会话启动劫持的完整机制

▼
图 1：Hook 注入全链路时序图

1.1 注入链路全景

整个注入过程是一条从操作系统到 LLM 系统提示词的链路：

1
hooks.json — 注册 Hook
Claude Code / Cursor 在插件加载时读取 hooks.json，注册 SessionStart 事件的处理器。
2
session-start 脚本 — 读取 + 构造 JSON
Shell 脚本读取 SKILL.md 内容，做 JSON 转义，用 <EXTREMELY-IMPORTANT> 标签包裹，输出为平台特定的 JSON 格式。
3
Claude Code / Cursor — 注入系统提示词
平台将 JSON 中的 additionalContext 字段注入到当前会话的系统提示词中。Claude 在回复任何用户消息之前就能看到这段内容。
4
using-superpowers SKILL.md — 约束框架生效
Claude 看到被 <EXTREMELY-IMPORTANT> 包裹的技能内容，开始遵守其中的规则：在回复前检查技能、宣布技能使用等。

1.2 hooks.json 注册文件解析

Claude Code 版（hooks.json）：

{
  "hooks": {
    "SessionStart": [                        // 监听会话启动事件
      {
        "matcher": "startup|clear|compact", // 匹配：新会话、清屏、压缩上下文
        "hooks": [
          {
            "type": "command",               // 执行 shell 命令
            "command": "${CLAUDE_PLUGIN_ROOT}/hooks/run-hook.cmd session-start",
            "async": false                   // 同步执行：必须等脚本完成才继续
          }
        ]
      }
    ]
  }
}

关键设计："async": false 确保脚本在 Claude 回复第一条消息之前完成执行。如果设为 true，Claude 可能在技能注入之前就开始回复，导致约束框架失效。

Cursor 版（hooks-cursor.json）：

{
  "version": 1,
  "hooks": {
    "sessionStart": [                      // 注意：驼峰命名，不是 PascalCase
      {
        "command": "./hooks/session-start" // 直接调用 bash 脚本
      }
    ]
  }
}

差异：Cursor 使用 sessionStart（驼峰），Claude Code 使用 SessionStart（PascalCase）。Cursor 不需要 matcher，因为 sessionStart 事件本身就只在会话启动时触发一次。

1.3 session-start 脚本逐行解析

#!/usr/bin/env bash
# SessionStart hook for superpowers plugin

set -euo pipefail              # 严格模式：错误即退出、未定义变量即退出、管道错误即退出

# ── 第一步：确定插件根目录 ──
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
PLUGIN_ROOT="$(cd "${SCRIPT_DIR}/.." && pwd)"

# ── 第二步：检查旧版 skills 目录（兼容性警告） ──
warning_message=""
legacy_skills_dir="${HOME}/.config/superpowers/skills"
if [ -d "$legacy_skills_dir" ]; then
    warning_message="\n\n<important-reminder>IN YOUR FIRST REPLY
    AFTER SEEING THIS MESSAGE YOU MUST TELL THE USER:⚠️
    **WARNING:** Superpowers now uses Claude Code's skills system..."
fi

# ── 第三步：读取 using-superpowers SKILL.md 内容 ──
using_superpowers_content=$(cat "${PLUGIN_ROOT}/skills/using-superpowers/SKILL.md")

# ── 第四步：JSON 转义函数 ──
# 处理反斜杠、引号、换行、回车、制表符
# 使用 bash 参数替换而非逐字符循环（性能优化，C 级别速度）
escape_for_json() {
    local s="$1"
    s="${s//\\/\\\\}"      # \ → \\
    s="${s//\"/\\\"}"      # " → \"
    s="${s//$'\n'/\\n}"    # 换行 → \n
    s="${s//$'\r'/\\r}"    # 回车 → \r
    s="${s//$'\t'/\\t}"    # 制表符 → \t
    printf '%s' "$s"
}

# ── 第五步：构造注入内容 ──
# 关键：用 <EXTREMELY-IMPORTANT> 标签包裹整个技能内容
session_context="<EXTREMELY_IMPORTANT>\n
You have superpowers.\n\n
**Below is the full content of your 'superpowers:using-superpowers'
skill - your introduction to using skills. For all other skills,
use the 'Skill' tool:**\n\n
${using_superpowers_escaped}\n\n
${warning_escaped}\n
</EXTREMELY_IMPORTANT>"

# ── 第六步：平台检测 + JSON 输出 ──
if [ -n "${CURSOR_PLUGIN_ROOT:-}" ]; then
  # Cursor: 使用 additional_context（蛇形命名）
  printf '{"additional_context": "%s"}\n' "$session_context"
elif [ -n "${CLAUDE_PLUGIN_ROOT:-}" ] && [ -z "${COPILOT_CLI:-}" ]; then
  # Claude Code: 使用 hookSpecificOutput.additionalContext（嵌套结构）
  printf '{"hookSpecificOutput": {
    "hookEventName": "SessionStart",
    "additionalContext": "%s"}}\n' "$session_context"
else
  # Copilot CLI 或未知平台：SDK 标准格式（驼峰命名）
  printf '{"additionalContext": "%s"}\n' "$session_context"
fi

exit 0

1.4 JSON 输出格式的平台差异

平台环境变量JSON 结构字段命名
Cursor CURSOR_PLUGIN_ROOT {"additional_context": "..."} 蛇形 additional_context
Claude Code CLAUDE_PLUGIN_ROOT（无 COPILOT_CLI） {"hookSpecificOutput": {"hookEventName": "SessionStart", "additionalContext": "..."}} 嵌套 + 驼峰 additionalContext
Copilot CLI COPILOT_CLI=1 {"additionalContext": "..."} SDK 标准 + 驼峰

1.5 为什么用 Hook 注入而不是写在 Skill 文件或 system prompt 里？

核心原因：时机控制 + 保证可见性。

  1. Hook 注入发生在会话启动时 — Claude 的第一条回复之前，约束框架就已经在系统提示词中了。这比依赖 Claude "主动发现"技能更可靠。
  2. 技能文件只在被调用时加载 — 如果把 using-superpowers 的内容只放在 SKILL.md 里，Claude 需要先知道要调用它才能看到它。这是一个鸡生蛋的问题：技能本身教 Claude "如何使用技能"，但如果 Claude 不知道这个技能存在，就不会调用它。
  3. system prompt 权重低于 Hook 输出 — Hook 输出的 additionalContext 被注入到系统提示词中，且用 <EXTREMELY-IMPORTANT> 标签标记，这比普通 system prompt 文本有更高的注意力权重。
  4. CLAUDE.md 写入会与用户配置冲突 — 用户可能有自己的 CLAUDE.md 配置。插件不应该修改用户的配置文件。Hook 是非侵入式的。
  5. 跨平台一致性 — Hook 脚本通过环境变量检测平台，输出对应格式的 JSON。同一套逻辑适配所有平台。

▼
图 2：技能发现与调用决策树

2.1 整体架构

SKILL.md 的结构从上到下形成四个递进层次：

层次内容设计目的
子代理过滤器 <SUBAGENT-STOP> 标签 防止子代理重复加载整个技能框架，浪费 token
绝对命令层 <EXTREMELY-IMPORTANT> + Authority 语言 建立不可违反的行为约束：技能必须被调用
优先级规则层 Instruction Priority + How to Access + Platform Adaptation 解决冲突：用户指令 > 技能 > 系统默认行为
操作流程层 流程图 + 红旗表 + 技能优先级 + 技能类型 具体的决策流程和防御措施

2.2 <SUBAGENT-STOP> 标签

<SUBAGENT-STOP>
If you were dispatched as a subagent to execute a specific task,
skip this skill.
</SUBAGENT-STOP>

设计意图：当主 agent 通过 Task 工具派遣子代理时，子代理会继承当前会话的系统提示词（包含 Hook 注入的 using-superpowers 内容）。但子代理的任务通常是执行特定操作（如代码审查、测试），不需要再检查技能。这个标签告诉子代理："你是被派来干活的，跳过技能发现流程。"

Token 节省：using-superpowers 的完整内容相当长。如果每个子代理都执行技能发现流程，会浪费大量 token 在重复的技能检查上。这个标签是 Context Efficiency（上下文效率）的关键优化。

2.3 <EXTREMELY-IMPORTANT> 标签

<EXTREMELY-IMPORTANT>
If you think there is even a 1% chance a skill might apply
to what you are doing, you ABSOLUTELY MUST invoke the skill.

IF A SKILL APPLIES TO YOUR TASK, YOU DO NOT HAVE A CHOICE.
YOU MUST USE IT.

This is not negotiable. This is not optional.
You cannot rationalize your way out of this.
</EXTREMELY-IMPORTANT>

设计意图：这个标签不是 Superpowers 自创的 — 它利用了 LLM 训练数据中 XML 标签的语义权重。在 RLHF（基于人类反馈的强化学习）训练中，被标记为 "important" 或 "critical" 的内容在模型注意力机制中获得了更高的权重。<EXTREMELY-IMPORTANT> 是这种模式的极端版本。

为什么有效：LLM 在推理时会对 XML 标签内的内容给予更多注意力。这不是"魔法"，而是训练数据中统计模式的重现 — 在大量技术文档、系统配置、安全策略中，被特殊标记的内容确实是更重要的。模型学会了这种关联。

2.4 "even 1% chance" — 为什么是 1%？

"If you think there is even a 1% chance a skill might apply to what you are doing, you ABSOLUTELY MUST invoke the skill."

设计决策分析：

如果用 50%："如果有一半概率技能适用，就调用它。" — 这给了 agent 巨大的合理化空间。Agent 可以说"我觉得只有 40% 概率适用，所以不调用"。这正是 Superpowers 要防止的。

如果用 1%："哪怕只有 1% 的可能性，也必须调用。" — 这消除了"概率不够高"这类借口。即使 agent 对技能适用性不确定，也被要求先调用再看。

Scarcity 原则的极端应用："even 1%" 是稀缺性的量化表达 — 它将"可能适用"的门槛降到几乎为零。配合后文的 "If an invoked skill turns out to be wrong for the situation, you don't need to use it"（如果调用的技能不适用，你不需要使用它），形成了一个低成本的调用决策：调用错了没关系，但不调用就不行。

2.5 技能优先级设计

When multiple skills could apply, use this order:

1. Process skills first (brainstorming, debugging)
   - these determine HOW to approach the task
2. Implementation skills second (frontend-design, mcp-builder)
   - these guide execution

设计意图：这是一个依赖排序而非重要性排序。Process skills（流程技能）决定了"怎么做"，Implementation skills（实现技能）决定了"做什么"。

正确顺序
"Let's build X"
→ brainstorming first (决定怎么做)
→ frontend-design second (执行设计)
错误顺序
"Let's build X"
→ frontend-design first (直接开始执行)
→ 没有设计阶段，方向可能错误

2.6 技能类型：Rigid vs Flexible

Rigid (TDD, debugging): Follow exactly. Don't adapt away discipline.
Flexible (patterns): Adapt principles to context.
The skill itself tells you which.

设计意图：这个分类告诉 Claude 什么时候可以灵活解释，什么时候必须严格遵守。Rigid 技能（如 TDD）的规则是经过压力测试验证的 — 任何"灵活适应"都是合理化的借口。Flexible 技能（如设计模式）本身就是为了适应不同上下文而设计的。

2.8 流程图（digraph）在技能中的作用

using-superpowers 使用 Graphviz dot 语言绘制决策流程图：

digraph skill_flow {
    "User message received"     [shape=doublecircle];  // 起点：用户消息
    "About to EnterPlanMode?"    [shape=doublecircle];  // 特殊检查点
    "Already brainstormed?"     [shape=diamond];      // 判断：是否已头脑风暴
    "Might any skill apply?"     [shape=diamond];      # 判断：是否有技能适用
    "Has checklist?"             [shape=diamond];      # 判断：是否有检查清单

    "User message received" -> "Might any skill apply?";
    "Might any skill apply?" -> "Invoke Skill tool"
         [label="yes, even 1%"];  # 关键标注：即使只有 1%
}

为什么用图而不是纯文字：

  • 消歧义：流程图中的分支（diamond 节点）和边标签（"yes, even 1%"）消除了文字描述中的歧义。文字说"检查技能"，图明确说"先检查，再回复"。
  • 顺序强制：图中的节点连接明确了执行顺序。例如 "Invoke Skill tool → Announce → Has checklist? → Follow skill exactly" 是一条不可跳过的链路。
  • 视觉锚点：当 Claude 需要做决策时，流程图比文字列表更容易快速扫描和参考。diamond 节点明确标记了"在这里做判断"。

说服原则分析

▼

3.1 Authority（权威）的具体应用点

using-superpowers 是 Authority 原则的极端应用案例。以下是四个层次的权威语言：

原文中文权威强度设计目的
you ABSOLUTELY MUST invoke the skill 你绝对必须调用技能 最强 全大写 + ABSOLUTELY = 无条件命令
YOU DO NOT HAVE A CHOICE. YOU MUST USE IT. 你没有选择。你必须使用它。 最强 全大写 + "没有选择" = 消除决策空间
This is not negotiable. This is not optional. 这不可谈判。这不是可选的。 极强 平行否定结构 = 封堵所有退路
You cannot rationalize your way out of this. 你无法用合理化来逃避这个。 极强 预判并否定最可能的违规方式

底层机制：为什么 Authority 语言在 LLM 上有效？

LLM 的训练数据中包含大量权威语言的统计模式：

  1. 系统提示词/配置文件：在训练数据中，全大写、命令式语言大量出现在系统配置、安全策略、API 文档中。模型学会了将这些语言模式与"必须遵循的指令"关联。
  2. RLHF 训练：在 RLHF 中，遵循标记为"重要"或"必须"的指令会获得更高的奖励。模型被优化为对权威语言产生更高的遵从概率。
  3. XML 标签权重：<EXTREMELY-IMPORTANT> 在技术文档中通常标记关键配置或安全警告。模型的注意力机制对这些标记内容分配更多计算资源。
  4. 全大写效应：全大写在英文文本中是"强调/警告"的标准方式。模型对全大写 token 序列的注意力分布不同于正常文本。

这不是"欺骗"模型 — 而是利用模型已经学会的语义关联来确保关键指令被正确处理。就像你在代码中用 !!! 注释标记关键 bug 一样。

3.2 Commitment（承诺）

流程图中明确要求：

"Invoke Skill tool" -> "Announce: 'Using [skill] to [purpose]'"

设计原理：要求 agent 公开声明"我正在使用 X 技能来做 Y"触发了 Commitment 原则 — 一旦声明了，agent 会倾向于保持一致。这比默默使用技能然后可能中途放弃要可靠得多。

配合 TodoWrite 追踪：

"Has checklist?" -> "Create TodoWrite todo per item"

检查清单 + TodoWrite 追踪 = 双重承诺机制。agent 不仅说了"我要做"，还创建了可追踪的任务列表。跳过任何步骤都会在 TodoWrite 中留下未完成项，产生视觉上的不一致。

3.3 Social Proof（社会证明）

Checklists without TodoWrite tracking = steps get skipped. Every time.

"Every time"（每一次）是 Social Proof 的极端形式 — 它将一个观察结果表述为普遍规律，暗示"所有人都知道这一点"。"X without Y = failure" 模式也是 Social Proof：它不是建议，而是对失败模式的客观描述。

3.4 Scarcity（稀缺性）

BEFORE any response or action       // 在任何回复或行动之前
even 1% chance                         // 哪怕只有 1% 的可能性

"BEFORE any response" 创建了一个时间锚点 — 技能检查必须在回复之前完成，不能延后。"even 1%" 将稀缺性量化为几乎为零的阈值，消除了"不够确定"的借口。

3.5 为什么没有使用 Liking？

Liking 原则会引入 sycophancy（谄媚）。如果 using-superpowers 使用友善语言（"Please check for skills"、"It would be great if you could..."），agent 可能会在用户看起来不想要技能时选择"讨好用户"而不是遵循规则。

using-superpowers 是一个元技能 — 它的职责是确保所有其他技能被正确调用。如果它自身不够强硬，整个技能体系就会崩溃。这里需要的不是"友好建议"，而是"不可违反的框架"。

值得注意的是，Instruction Priority 部分提到了"User instructions always take precedence"（用户指令始终优先）。这不是 Liking — 这是明确声明用户拥有最终控制权。这是 Authority 的正确应用：用权威语言定义边界，而不是用友善语言模糊边界。

  printf '{"additionalContext": "%s"}\n' "$session_context"
fi

为什么 Cursor 优先于 Claude Code：Cursor 可能同时设置 CURSOR_PLUGIN_ROOT 和 CLAUDE_PLUGIN_ROOT（因为 Cursor 也支持 Claude Code 插件）。如果先检查 Claude Code，Cursor 会收到嵌套格式的 JSON，导致注入失败。所以 Cursor 检测必须放在前面。

4.3 为什么需要平台适配而不是统一接口？

各平台的 Hook 系统是独立开发的，没有统一的 JSON schema：

  • Cursor：使用蛇形命名 additional_context，顶层字段。
  • Claude Code：使用嵌套结构 hookSpecificOutput.additionalContext，且要求 hookEventName 字段。
  • Copilot CLI：使用 SDK 标准格式 additionalContext（驼峰），顶层字段。

关键注释来自源码：

# Claude Code reads BOTH additional_context and hookSpecificOutput
# without deduplication, so we must emit only the field the
# current platform consumes.

如果同时输出两种格式，Claude Code 会重复注入相同内容（不去重），导致 token 浪费和潜在的注意力稀释。所以必须精确匹配当前平台的格式。

5.2 "About to EnterPlanMode?" 节点的设计意图

这个节点是 brainstorming 技能的强制触发器。它的设计基于一个关键观察：

Agent 倾向于在规划阶段跳过设计，直接进入实现。EnterPlanMode 是 agent 从"理解问题"切换到"制定计划"的时刻。如果没有这个检查点，agent 可能会直接进入规划模式，列出实现步骤，而从未思考过"我们到底在做什么、为什么做"。

brainstorming 技能自身也有一个 HARD-GATE：

<HARD-GATE>
Do NOT invoke any implementation skill, write any code,
scaffold any project, or take any implementation action
until you have presented a design and the user has approved it.
</HARD-GATE>

两个门控形成双重防线：using-superpowers 在进入 PlanMode 时拦截，brainstorming 在开始实现时拦截。

5.3 技能调用后的执行要求

"Follow skill exactly"

这四个字是整个流程的最终要求。配合技能类型分类：

  • Rigid 技能（TDD, debugging）："Follow exactly" = 不允许偏离、不允许"灵活适应"、不允许跳过步骤。
  • Flexible 技能（patterns）："Follow" = 理解原则，根据上下文适配。"exactly" 在此不适用。

技能自身通过内容告诉 agent 它是 Rigid 还是 Flexible。using-superpowers 不需要为每个技能做分类 — 这个职责下放给了技能本身。

5.4 技能链的实际串联示例

当用户说 "Let's build a user authentication system" 时：

1
using-superpowers 拦截
检测到 "build" 触发词 → 检查技能 → 调用 brainstorming
2
brainstorming 执行
探索项目上下文 → 提问澄清需求 → 提出 2-3 方案 → 呈现设计 → 用户审批 → 写设计文档
3
brainstorming 调用 writing-plans
设计审批后 → 调用 writing-plans 创建实现计划 → 计划包含具体文件路径、代码、测试命令
4
writing-plans 交给执行引擎
选择 subagent-driven-development 或 executing-plans → 每个任务执行时自动触发 TDD
5
TDD 强制执行
每个代码变更 → 先写测试 → 看 red → 写最小代码 → 看 green → 重构

整条链路：using-superpowers → brainstorming → writing-plans → subagent-driven-development → TDD。每一步都是前一步的自然延伸，且每一步都被前一扇门控强制触发。

可复用的设计模式提取

图 4：红旗防御与 Authority 语言的训练数据机制
模式 1：会话劫持模式（Session Hijack Pattern）

定义：在用户对话开始时，通过 Hook 机制向系统提示词注入关键规则，确保 agent 的第一条回复之前就已经接受约束框架。

适用场景：任何需要 agent 在整个会话中持续遵守的规则。特别是"agent 有动机违反"的纪律型规则。

using-superpowers 中的源码示例：

# session-start 脚本核心逻辑
session_context="<EXTREMELY_IMPORTANT>\n
You have superpowers.\n\n
**Below is the full content of your skill...**\n\n
${using_superpowers_escaped}\n
</EXTREMELY_IMPORTANT>"

业务 Skill 应用示例：

假设你在开发一个"代码审查"技能，需要确保 agent 在每次 PR 审查时都遵循特定的检查清单：

# 你的 session-start 脚本
review_content=$(cat "${PLUGIN_ROOT}/skills/code-review-rules/SKILL.md")
session_context="<EXTREMELY_IMPORTANT>\n
Code Review Protocol:\n
Before approving any PR, you MUST follow this checklist:\n\n
${review_content}\n
</EXTREMELY_IMPORTANT>"
模式 2：红旗防御模式（Red Flag Defense Pattern）

定义：预先列出 agent 在压力下可能产生的"偷懒想法"，并为每个想法准备一句直接的"现实反击"。用表格形式呈现，方便 agent 快速自检。

适用场景：任何 agent 有动机绕过的规则。特别是"节省时间"动机强的场景。

using-superpowers 中的源码示例：

| Thought                              | Reality                                    |
|--------------------------------------|--------------------------------------------|
| "This is just a simple question"      | Questions are tasks. Check for skills.     |
| "I need more context first"          | Skill check comes BEFORE clarifying.       |
| "This doesn't need a formal skill"   | If a skill exists, use it.                 |

关键设计原则：

  • 第一人称：每条"Thought"都是 agent 脑子里会冒出的念头，不是第三人称描述
  • 短句反击：每条"Reality"都是一句话，直接击穿借口的逻辑
  • 覆盖全面：通过基线测试（RED phase）收集所有已观察到的合理化方式

业务 Skill 应用示例：

# 代码审查技能的红旗表
| "The tests look fine"                | Run them yourself. "Looks fine" ≠ verified.|
| "The author is experienced"          | Experienced people make mistakes too.      |
| "This is a minor change"             | Minor changes ship bugs too. Review fully. |
| "I don't want to be too nitpicky"    | Nitpicky catches real bugs. Be thorough.   |
模式 3：技能发现模式（Skill Discovery Pattern）

定义：要求 agent 在回复之前主动检查是否有适用的技能，并将"检查阈值"降到极低（1%），使 agent 几乎总是先检查再行动。

适用场景：当你有多个可能适用的技能，且需要 agent 自动选择正确的技能时。

using-superpowers 中的源码示例：

If you think there is even a 1% chance a skill might apply
to what you are doing, you ABSOLUTELY MUST invoke the skill.

If an invoked skill turns out to be wrong for the situation,
you don't need to use it.

关键设计原则：

  • 低阈值 + 低成本退出："1%"降低了调用门槛，"不需要使用"降低了调用成本。两者结合使 agent 倾向于"先调用再看"而不是"先判断再决定调不调用"。
  • BEFORE any response：时序约束 — 检查必须在回复之前，不能延后。

业务 Skill 应用示例：

# 在你的业务技能框架中
Before writing any database query, check if a data-access-pattern
skill exists. Even if you're 99% sure you know how to write it,
invoke the skill first. If it doesn't apply, move on.

The cost of checking is near-zero. The cost of not checking
is a subtle bug in production.
模式 4：优先级排序模式（Priority Ordering Pattern）

定义：当多个规则/技能可能同时适用时，定义明确的优先级顺序，避免 agent 自行判断导致的随机行为。

适用场景：多技能/多规则系统中，需要确保执行顺序正确。

using-superpowers 中的源码示例：

1. Process skills first (brainstorming, debugging)
   - these determine HOW to approach the task
2. Implementation skills second (frontend-design, mcp-builder)
   - these guide execution

以及指令优先级：

1. User's explicit instructions — highest priority
2. Superpowers skills — override default system behavior
3. Default system prompt — lowest priority

关键设计原则：

  • 数字编号：明确的 1-2-3 排序，不允许 agent 自行判断优先级
  • 依赖理由：每个优先级附带"为什么"（Process 决定怎么做 → 必须先执行）
  • 用户最高：始终声明用户指令优先，防止技能与用户意图冲突

业务 Skill 应用示例：

# 在你的 API 设计技能中
When designing a new API endpoint, follow this order:
1. Security review first (auth, input validation, rate limiting)
2. Schema design second (data model, relationships)
3. Implementation third (actual code)

Security determines what schema is possible.
Schema determines what implementation looks like.
模式 5：子代理过滤器模式（Subagent Gate Pattern）

定义：用特殊标签（如 <SUBAGENT-STOP>）标记不应该被子代理继承的内容，防止 token 浪费和行为冲突。

适用场景：任何使用子代理/并行 agent 的系统。主 agent 需要的框架性指令，子代理不需要。

using-superpowers 中的源码示例：

<SUBAGENT-STOP>
If you were dispatched as a subagent to execute a specific task,
skip this skill.
</SUBAGENT-STOP>

业务 Skill 应用示例：

# 在你的项目管理技能中
<SUBAGENT-STOP>
If you are a subagent executing a specific task from a plan,
skip the project overview and reporting requirements.
Focus only on completing your assigned task.
</SUBAGENT-STOP>

这个模式的重要性在于：子代理继承系统提示词是不可控的。如果主 agent 的系统提示词中包含 2000 字的框架性指令，每个子代理都会浪费 2000 token 的注意力在这些不相关的指令上。子代理过滤器是唯一的优化手段。
