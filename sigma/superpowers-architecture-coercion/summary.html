01 会话启动劫持：Hook 作为"第一印象"攻击向量 80% 已掌握
源码
hooks/session-start using-superpowers/SKILL.md
核心洞察
三层架构实现会话劫持：Hook 脚本提供程序性保证（模型必定看到）， <EXTREMELY_IMPORTANT> 标签提供注意力加权（训练数据中 "extremely important" 后面的内容需要优先处理）， Authority 语言提供行为锁定（训练数据中 "YOU MUST" 后面跟着服从行为）。 三层叠加 = 模型"被迫"在会话开始时就接受约束框架。
关键短语
YOU ABSOLUTELY MUST YOU DO NOT HAVE A CHOICE This is not negotiable
学习过程
Q: 为什么用 Hook 注入而不是写成 Skill 等模型自己调用？
程序性保证 > 提示性建议。Skill 是模型可以选择调用的，Hook 是模型无法绕过的。
Q: 把 Authority 语言改成 "Please consider" 会怎样？
同样的注入时机，行为结果不同。因为训练数据中 "please consider" 后面跟着的是选择性行为，而非强制行为。
纠正的误区
误以为标签名有内在"高权重"。实际是训练数据统计模式——"extremely important" 在训练数据中与高优先级内容高频共现。
02 Prompt 中的说服心理学：权威、承诺、稀缺性 80% 已掌握
源码
writing-skills/persuasion-principles.md
核心洞察
三原则协同锁定模型行为：
Authority（外部规则："YOU MUST"）→ 在被注入时生效
Commitment（内部锁定：模型自己输出"我正在使用 TDD"）→ 模型的自输出成为后续行为的约束上下文（自生成上下文创造自洽压力）
Social Proof（规范强化："每次都这样，不这样做就是失败"）→ 贯穿全程

Liking 被标记为 Avoid，因为它引入竞争性目标——当"让用户开心"和"遵守流程"冲突时，模型会选择前者。
学习过程
Q: 为什么 TDD 铁律里加 "作为你的好伙伴，我希望你能先写测试" 会削弱强制力？
Liking 原则引入竞争性目标。模型的目标从"遵守流程"变成"让用户喜欢我"。当用户赶时间说"跳过测试吧"，Liking 驱动的模型会顺从，Authority 驱动的模型会拒绝。
Q: 为什么"说出来"（Commitment 宣布）能增强遵守概率？
模型自己输出的内容会成为后续生成的上下文。声明"我正在使用 TDD"后，后续 token 预测会倾向与声明一致的行为。外部压力 + 内部自洽 + 规范锚定 = 三层锁定。
03 反合理化表格与红旗检测：预判模型的借口 80% 已掌握
源码
systematic-debugging 红旗表 TDD 红旗表 verification-before-completion 合理化预防表
核心洞察
设计模式：不阻止"偷懒的想法"本身，而是预判它一定会出现，然后预装 When/Then 触发动作。

模型找借口不是因为有 Commitment，而是因为训练数据中"跳过测试时先给自己找个理由"是人类程序员的高频行为模式。反合理化表格把"借口→反驳"配对预先写进上下文，当模型生成借口时，上下文中已有反驳，后续 token 自然跟着反驳走。
关键短语
Simple code breaks. Test takes 30 seconds. STOP. Return to Phase 1. Delete it. Start over.
设计模式
"If you catch yourself thinking [X]" 比 "Don't think about X" 更有效——因为它承认模型一定会产生这个想法（训练数据模式），然后给出想法出现后的触发动作。这是 Implementation Intentions：Clear triggers + required actions = automatic execution。
纠正的误区
误以为找借口的行为源于 Commitment 原则。实际是训练数据中人类逃避行为模式的复现。借口→反驳配对预加载到上下文后，模型的自生成借口文本会触发预装的反驳。

04 头脑风暴：brainstorming 如何在写代码前强制收敛需求 80% 已掌握
源码
skills/brainstorming/SKILL.md
核心洞察
双重门控设计：第一次批准锁定方向，第二次批准捕获文档化偏差。

从口头到书面不是"转录"而是"转化"——模型会填补未讨论的空白、把隐含的东西显式化、增加双方没约定的细节。spec 文档是下游整条链的契约/根门控，spec 错了后面全错。
学习过程
Q: 为什么需要两次用户批准而不是一次？
第一次在设计展示后——锁定方向。第二次在 spec 文档审阅后——口头→书面是转化，必然产生偏差（填补空白、显式化隐含内容、增加未约定细节）。如果只在最后审批一次，可能发现整个过程都在错误方向。
关键约束
<HARD-GATE> Do NOT invoke any implementation skill until design approved </HARD-GATE>
05 计划即契约：writing-plans 如何创建不可违背的执行规格 80% 已掌握
源码
skills/writing-plans/SKILL.md
核心洞察
链式传递中的"零上下文"原则：每个交接点的接收方都是零上下文启动（全新 subagent），因此每个交接输出必须完全自包含。

Plan 中每个 Step 必须包含完整代码和预期命令输出，因为执行 Task N 的 subagent 从未被邀请参加 brainstorming，没看过 spec 讨论。它的上下文里只有 plan 文档中 Task N 那几段文字。
架构原则
三个 No Placeholders 背后是同一原则：任何依赖隐含上下文的交接都是失败点。
06 计划执行交接：subagent 双模式 75% 已掌握
源码
skills/executing-plans/SKILL.md skills/subagent-driven-development/SKILL.md
核心洞察
两种执行模式：Inline Execution（单会话批量执行，有检查点）和 Subagent-Driven（每个 Task 派发全新 subagent + 两阶段审查：先 spec compliance 再 code quality）。

spec compliance 必须先于 code quality——如果代码没实现 spec 要求的功能，代码写得再优雅也没意义。这是顺序门控原则。
模型选择策略
机械实现任务用便宜模型，集成判断任务用标准模型，架构/设计/审查任务用最强模型。这解释了为什么不信任 subagent 的 "success" 报告——它可能是能力最弱的模型。
纠正的误区
误以为 spec→quality 审查顺序和 worktree 基线验证的共同原则是 Commitment。实际是顺序门控（Sequential Gating）：先确认"做了正确的事"，再确认"把事做正确"。
07 Git Worktree 隔离：分支物理隔离 75% 已掌握

核心洞察
创建隔离工作区后必须先跑一遍测试确认基线是干净的才能开始开发。如果基线测试就是红的，改完代码后跑测试还是红的——永远分不清是你搞坏的还是本来就坏的。

这和 spec→quality 审查顺序是同一个原则：先建立已知良好基线，再在此基础上构建。

08 TDD 铁律：红-绿-重构作为强制门控 85% 已掌握
源码
skills/test-driven-development/SKILL.md
核心洞察
"Delete it. Start over." 不是浪费，而是消除偏见源。

保留代码再补测试，测试会被实现绑架——变成"验证现有行为"而不是"定义期望行为"。Tests-after answer "What does this do?" Tests-first answer "What should this do?"

这又是干净基线原则的变体：和 worktree 先验证基线、审查先 spec 后 quality 是同一个模式。
学习过程
Q: 为什么要求 delete 而不是 keep it and write tests around it？
保留代码再补测试，模型会为了偷懒编写让现有代码能通过的测试，只测试能正常执行的功能，无法保证测试有效性。系统性风险：tests biased by implementation。
09 完成前验证："禁止虚假成功"防御与 Gate Function 80% 已掌握
源码
skills/verification-before-completion/SKILL.md
Gate Function 五步
1. IDENTIFY — 什么命令能证明这个 claim？
2. RUN — 执行完整命令
3. READ — 读取完整输出，检查 exit code，统计失败数
4. VERIFY — 输出是否确认 claim？
5. CLAIM — 只有这时才能声称结果
Skip any step = lying, not verifying
核心洞察
"success" 是一个 claim，不是一个 fact。VCS diff 是 evidence。
不信任 subagent 的 "success" 报告——执行 Task 的 subagent 可能是能力最弱的模型，它说 success 时可能根本没意识到自己漏了东西。
关键短语
Evidence before claims, always. Skip any step = lying, not verifying
10 系统化调试：四阶段根因分析与架构质疑机制 80% 已掌握
源码
skills/systematic-debugging/SKILL.md defense-in-depth.md root-cause-tracing.md
核心洞察
3 次修复阈值的本质不是计数而是模式识别：
1 次是失误，2 次是运气差，3 次说明每次修复都在不同位置产生新问题。这时已经不是在 debug，而是在给一个根本性的结构缺陷打补丁。

判断标准不是"修了 3 次没好"，而是"每次修复都暴露新的耦合或状态问题"。数量是表象，模式才是信号。
关键约束
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST Violating the letter of this process is violating the spirit of debugging.

11 设计你自己的强制系统：全链路综合与实践 72% 已掌握
五层防御体系
L0 模型思考之前 Hook 注入 — 保证模型看到规则
L1 模型开始工作时 Authority 铁律 + Commitment 宣布 — 锁定行为
L2 模型想偷懒时 反合理化表格 — 堵死借口
L3 模型想跳步时 顺序门控 + When/Then 触发器 — 强制走完流程
L4 模型说"做完了"时 Gate Function — 必须给证据
诊断框架
不同失败动机对应不同防御层：
跳步（流程问题）→ Layer 3 门控
找借口（心理问题）→ Layer 2 反合理化
假完成（证据问题）→ Layer 4 Gate Function

核心能力差距：能识别单个模式但难以自发组织成分层防御体系。诊断失败动机并选择防御层的能力需要持续练习。
纠正的误区
倾向于把所有问题统一归到"流程没走对"（Layer 1/3），忽略模型在不同阶段的失败动机不同。正确做法：先诊断失败动机（跳步/找借口/假完成），再选择对应防御层。

误区记录

M1 过度套用 Commitment 框架（第 1 次）
误以为反合理化表格中"找借口"的行为源于 Commitment 原则。

根因：过度套用 Commitment 框架，未理解借口的本质是训练数据中人类逃避行为模式的复现。

纠正：借口→反驳配对预加载到上下文后，模型自生成的借口文本会触发预装的反驳，这是 Authority 的一种特殊形式（预先堵死逃跑路线），不是 Commitment。
M2 过度套用 Commitment 框架（第 2 次）
误以为 spec→quality 审查顺序和 worktree 基线验证的共同原则是 Commitment。

根因：再次过度套用 Commitment 框架，将所有机制都归结为承诺/自洽。

纠正：共同原则是顺序门控（Sequential Gating）——先确认"做了正确的事"，再确认"把事做正确"。没有干净基线，后续所有验证都是无效的。
M3 统一归因：把所有问题都归到"流程没走对"
在综合实践中，将跳步、找借口、假完成全部归因到 Layer 1/3（流程/锁定），未区分模型在不同阶段的失败动机。

根因：未区分跳步（流程问题）vs 找借口（心理问题）vs 假完成（证据问题）。

纠正：诊断失败动机，选择对应防御层——跳步→Layer3门控，找借口→Layer2反合理化，假完成→Layer4 Gate Function。

后续练习方向

1. 用实际案例做诊断练习
找 3 个你团队实际遇到过的"模型不听话"案例，对每个用五层防御框架做诊断：失败对应哪一层缺失？
2. 设计完整的 Layer 0-4 规则
针对诊断结果，写一个完整的分层防御规则，确保每一层都有对应机制。不要只写一两层——五层全覆盖。
3. 实际测试并迭代
把规则给模型用，观察它是否在预期的层被拦截了。如果没有，回到五层框架重新诊断，修改规则，再次测试。
