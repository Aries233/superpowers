01

设计哲学与核心约束

▼

1.1 这个技能解决什么问题？

Brainstorming 解决的核心问题是：

AI agent（和人类开发者）拿到需求后直接动手实现，跳过了"想清楚要做什么"的阶段。

"直接动手"的问题不在于"做得不对"，而在于"做的可能不是该做的事"。当你没有在设计阶段暴露假设和不确定性时，这些隐藏的问题会在实现阶段爆发 — 而那时的修改成本是设计阶段的 10-100 倍。

技能的核心定位用一句话概括：

Help turn ideas into fully formed designs and specs through
natural collaborative dialogue.
通过自然的协作对话，将想法转化为完整的设计和规格文档。

1.2 Anti-Pattern："This Is Too Simple To Need A Design"

## Anti-Pattern: "This Is Too Simple To Need A Design"

Every project goes through this process. A todo list, a
single-function utility, a config change — all of them.
"Simple" projects are where unexamined assumptions cause the
most wasted work. The design can be short (a few sentences for
truly simple projects), but you MUST present it and get approval.

为什么强制所有项目都走这个流程？

这是经过实证观察得出的规则。"简单"项目恰恰是最容易出问题的：

  • 简单的项目最容易被低估 — "不就是加个字段吗"的假设可能忽略了数据库迁移、API 兼容性、缓存失效等连锁影响
  • 简单的项目最容易被跳过设计 — 因为"看起来不需要"，所以不检查假设，导致隐藏的依赖和约束在实现时才被发现
  • 简单的项目设计文档也很短 — 技能明确说 "The design can be short (a few sentences)"，门槛极低，但没有例外

1.3 <HARD-GATE> 标签的约束力

<HARD-GATE>
Do NOT invoke any implementation skill, write any code,
scaffold any project, or take any implementation action
until you have presented a design and the user has approved it.
This applies to EVERY project regardless of perceived simplicity.
</HARD-GATE>

它禁止了什么：

  • 调用任何实现类技能（frontend-design、mcp-builder 等）
  • 写任何代码
  • 脚手架化任何项目
  • 采取任何实现行动

为什么必须禁止：这是 brainstorming 技能存在的全部意义。如果允许 agent 在设计完成前"先写一点代码试试"，agent 就会绕过整个设计流程。这不是理论假设 — 这是 LLM 的已知行为模式：当有更"直接"的路径（写代码）时，agent 会选择跳过"间接"的路径（设计讨论）。

02

九步 Checklist 深度拆解

▼
图 1：九步 Checklist 流程图
Step 1: Explore project context — 探索项目上下文

做什么：检查现有文件、文档、最近的 git commits，理解项目当前状态。

Check out the current project state first (files, docs, recent commits)

为什么不能跳过：不了解现有代码库就提出设计方案，等于闭门造车。你的设计可能与现有架构冲突、重复已有功能、或忽略已建立的模式。

额外设计决策 — 范围评估：

Before asking detailed questions, assess scope: if the request
describes multiple independent subsystems (e.g., "build a platform
with chat, file storage, billing, and analytics"), flag this
immediately. Don't spend questions refining details of a project
that needs to be decomposed first.

这是一个早期退出优化：如果需求范围过大，先拆分再设计，不要在错误粒度上浪费时间。

跳过的后果：设计方案与现有代码不兼容、重复造轮子、忽略已解决的边界问题。

Step 2: Offer visual companion — 提供可视化伴侣

做什么：如果即将讨论的话题涉及视觉内容（mockup、布局、架构图），主动提供浏览器端的可视化伴侣。

为什么是独立消息：

This offer MUST be its own message. Do not combine it with
clarifying questions, context summaries, or any other content.
The message should contain ONLY the offer above and nothing else.

如果将 offer 与其他内容混在一起，用户可能忽略 offer 或被其他信息分散注意力。独立消息 = 100% 注意力 = 明确的 yes/no 决策。

跳过的后果：需要视觉展示的问题（如布局选择）只能用文字描述，用户难以理解和选择。

Step 3: Ask clarifying questions — 一次一个澄清问题

做什么：一次问一个问题来细化需求，聚焦于目的、约束、成功标准。

为什么"一次只问一个问题"：

认知负荷控制：多个问题同时出现时，用户（和 agent）倾向于只回答第一个而忽略其余。一次一个问题确保每个问题都得到充分思考。

对话自然性：自然的人类对话就是一问一答。强制这种模式使整个流程感觉像"协作讨论"而非"审讯"。

依赖链控制：问题 3 的答案可能改变问题 4 的内容。一次一个问题允许你根据前一个答案调整后续问题。

为什么"multiple choice preferred"：

Prefer multiple choice questions when possible, but open-ended
is fine too

多选题降低了用户的回答成本，同时给 agent 提供了结构化的信息。但不是所有问题都适合多选 — 当你需要开放式探索时，开放式问题更合适。

跳过的后果：基于未经验证的假设进行设计，实现时发现需求理解错误。

Step 4: Propose 2-3 approaches — 提出 2-3 个方案

做什么：提出 2-3 个不同的实现方案，附带权衡分析和你的推荐。

Propose 2-3 different approaches with trade-offs
Present options conversationally with your recommendation
and reasoning
Lead with your recommended option and explain why

为什么是 2-3 个而不是 1 个：

提出 2-3 个方案
A) REST API + PostgreSQL
   简单、成熟、易于调试
B) GraphQL + PostgreSQL
   灵活查询、减少过度获取
C) tRPC + SQLite
   全栈类型安全、零 API 代码

推荐 A，因为...（权衡分析）
只提出 1 个方案
我们用 REST API + PostgreSQL。

（没有对比，没有权衡，
用户无法评估这是否最优）
  • 强制权衡思考：当你需要解释"为什么推荐 A 而不是 B"时，你被迫思考真正的 trade-off
  • 让用户参与决策：用户可能了解你不知道的约束（如团队技术栈偏好、性能要求）
  • 防止确认偏误：只看一个方案时，你倾向于认为它是最好的。多个方案打破这种偏误

"Lead with your recommended option"：先说推荐，再说理由。这比先列举所有选项再最后说推荐更高效 — 用户可以直接聚焦在你认为最好的方案上，同时了解其他选项为什么被排除。

跳过的后果：陷入第一个想到的方案，错过更优解，用户没有参与感。

Step 5: Present design — 分段呈现设计

做什么：分段呈现设计，每段之后获取用户批准。覆盖：架构、组件、数据流、错误处理、测试。

Scale each section to its complexity: a few sentences if
straightforward, up to 200-300 words if nuanced
Ask after each section whether it looks right so far
Cover: architecture, components, data flow, error handling, testing

为什么分段呈现：

  • 增量验证：每段获得批准后才进入下一段。如果架构方案被否决，不需要浪费时间为错误的架构写组件设计
  • 复杂度适配：简单的部分"几句话"就够，复杂的部分最多 200-300 词。不是所有部分都需要同等详细度
  • 允许回退："Be ready to go back and clarify if something doesn't make sense"

跳过的后果：用户只看到完整设计无法参与逐段讨论，可能在最后才提出根本性的反对意见。

Step 6: Write design doc — 写设计文档

做什么：将经过口头验证的设计写入 spec 文档并提交到 git。

Write the validated design (spec) to
docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md
(User preferences for spec location override this default)
Commit the design document to git

为什么必须写文档（而不只是口头确认）：

口头 → 书面是"转化"而非"转录"。将口头讨论转化为文档时，模型会：

  • 填补未讨论的空白：口头讨论中跳过的细节必须在文档中显式化
  • 显式化隐含内容："大家都知道"的假设必须写出来
  • 增加未约定细节：文档需要完整到能让零上下文的工程师执行

这些"转化"过程中产生的新内容可能包含错误或与用户意图不符的假设。因此必须让用户在文档化之后再次审批（Step 8）。

跳过的后果：下游实现时没有可参考的权威文档，不同执行者对"要做什么"产生不同理解。

Step 7: Spec self-review — 规格自检

做什么：用四项检查自审 spec 文档，发现问题立即修复。

（详见第四部分）

Step 8: User reviews written spec — 用户审阅规格文档

做什么：让用户审阅写好的 spec 文档，等待用户确认。

After the spec review loop passes, ask the user to review
the written spec before proceeding:

"Spec written and committed to <path>. Please review it and
let me know if you want to make any changes before we start
writing out the implementation plan."

Wait for the user's response. If they request changes, make them
and re-run the spec review loop. Only proceed once the user
approves.

为什么这是第二个门控：Step 5 的口头批准和 Step 8 的文档审批是不同的事。口头讨论中用户同意的是"大方向"，文档中可能包含了口头讨论时未涉及的具体决策。用户必须看到这些"转化产生的新内容"并确认它们是正确的。

跳过的后果：文档中的错误假设直接流入实现计划，导致实现返工。

Step 9: Transition to implementation — 过渡到实现

做什么：调用 writing-plans 技能创建实现计划。

Invoke the writing-plans skill to create a detailed
implementation plan
Do NOT invoke any other skill. writing-plans is the next step.

为什么只允许 writing-plans：（详见第六部分）

双重门控机制

▼
图 2：双重门控与信息转化三阶段
图 3：口头 → 书面的信息转化分析（三种偏差类型）
G1
Gate 1：口头设计展示 → 用户批准
在 Step 5 中，agent 分段呈现设计方案，每段之后询问 "does it look right so far?"。用户必须明确批准设计后才能进入 Step 6（写文档）。
"Present design sections" -> "User approves design?"
"User approves design?" -> "Write design doc" [label="yes"]
"User approves design?" -> "Present design sections" [label="no, revise"]
G2
Gate 2：规格文档 → 用户审阅
在 Step 8 中，agent 写完 spec 后请用户审阅。用户批准后才能进入 Step 9（实现计划）。如果用户要求修改，回到 Step 6 重新写文档并再次自检。
"User reviews spec?" -> "Write design doc" [label="changes requested"]
"User reviews spec?" -> "Invoke writing-plans skill" [label="approved"]

核心洞察：口头 → 书面是"转化"而非"转录"

这是 brainstorming 技能中最精妙的设计洞察。

很多人认为"写文档"只是把讨论过的内容记录下来。但实际发生的是转化：

  • 填补空白：口头讨论中说"我们用 REST API"，文档中必须写明具体的 endpoint、请求格式、错误码、认证方式等从未讨论过的细节
  • 显式化隐含：口头讨论中"大家都知道"的上下文在文档中必须显式写出，因为文档的读者（子代理、未来的开发者）没有这些隐含上下文
  • 增加细节：文档需要达到"零上下文的工程师可以执行"的粒度，这必然引入口头讨论中不存在的新决策

因此，Gate 2 不可省略。用户在 Gate 1 批准的是口头设计方案。Gate 2 需要用户确认"文档化过程中产生的新内容"是否正确。这不是重复审批 — 是对不同层次内容的审批。

Spec 文档作为下游链的"根契约"

spec 文档是整条技能链的单一真相源：

brainstorming (spec) → writing-plans (plan) → executing-plans (code)
     ↓                      ↓                        ↓
   根契约               实现计划                代码实现

如果 spec 错了，plan 就会基于错误的假设，code 就会实现错误的功能。错误在链路下游被发现时的修复成本呈指数增长。这就是为什么 brainstorming 在 spec 上投入如此多的防御层（双重门控 + 自检 + 用户审阅）。

04

Spec Self-Review 机制

▼

4.1 四项自检

检查项做什么为什么重要
Placeholder scan
占位符扫描		 查找 "TBD"、"TODO"、不完整的段落、模糊的需求描述 占位符意味着"这里还没想清楚"，实现时必然出错
Internal consistency
内部一致性		 检查各部分之间是否矛盾 — 架构描述与功能描述是否匹配 矛盾意味着需求有冲突，实现时必须选择一个方向
Scope check
范围检查		 这个 spec 是否聚焦到单个实现计划？还是需要拆分为多个子项目？ 过大的 spec 会导致过长的实现计划，增加执行风险
Ambiguity check
歧义检查		 是否有需求可以被两种不同方式理解？如果是，选择一种并显式说明 歧义需求 = 不同的执行者会做出不同的实现

4.2 为什么是"自检"而不是"让审查者检查"？

After writing the spec document, look at it with fresh eyes:
...
Fix any issues inline. No need to re-review — just fix and move on.

效率决策：Self-review 是 agent 自己执行的四项检查，不需要派遣子代理。这四项检查是机械性的（搜索占位符、检查矛盾、评估范围、识别歧义），不需要人类的审美判断或业务知识。agent 自己就能可靠完成。

但 brainstorming 也提供了一个可选的子代理审查模板（spec-document-reviewer-prompt.md），用于更严格的外部审查。

4.3 "Fix any issues inline. No need to re-review" — 为什么只修不需要重新审？

因为四项检查是增量修复型的：发现占位符就填充内容，发现矛盾就修正一方，发现歧义就明确语义。这些修复不会引入新的问题类别。如果修复本身引入了新矛盾，那会在下一步（Gate 2: 用户审阅）被发现。

4.4 与 writing-plans 中 Self-Review 的对比

维度brainstorming Self-Reviewwriting-plans Self-Review
检查对象spec 文档（设计规格）plan 文档（实现计划）
检查项4 项（占位符、一致性、范围、歧义）3 项（覆盖度、占位符、类型一致性）
执行者agent 自身agent 自身
修复策略"Fix inline, no need to re-review""Fix inline, no need to re-review"
后续门控Gate 2: 用户审阅 spec无（直接进入执行）

关键差异：brainstorming 的 self-review 后有用户审阅门控（Gate 2），writing-plans 的 self-review 后没有。因为 writing-plans 的输入（spec）已经过了双重门控验证，质量有保障。

05

Visual Companion 设计决策

▼

5.1 Offer 必须是独立消息

This offer MUST be its own message. Do not combine it with
clarifying questions, context summaries, or any other content.
The message should contain ONLY the offer above and nothing else.
Wait for the user's response before continuing.

设计原理：当一条消息包含多个信息时，用户倾向于只回应最后一条或最紧急的一条。Visual companion 的 offer 需要明确的 yes/no 决策。如果与其他内容混在一起，用户可能忽略 offer 或做出不明确的回应。独立消息 = 100% 注意力 = 干净的决策。

5.2 Per-question decision — 逐题决策

Even after the user accepts, decide FOR EACH QUESTION whether
to use the browser or the terminal.
The test: would the user understand this better by seeing it
than reading it?

为什么不是"接受了就用浏览器"：接受了 companion 不意味着每个问题都需要可视化。有些问题是概念性的（"这个功能的目的是什么？"），用文字更高效。接受 companion 只是获得了使用浏览器的权限，不是承诺每个问题都用浏览器。

5.3 "UI 问题不一定是视觉问题"的设计智慧

A question about a UI topic is not automatically a visual question.
"What does personality mean in this context?" is a conceptual
question — use the terminal.
"Which wizard layout works better?" is a visual question —
use the browser.

这个区分体现了内容类型 vs 话题类型的思考：

  • 话题是 UI，但问题是概念性的："我们这个产品的设计风格应该是什么？" — 这是概念选择，多选题更合适
  • 话题是 UI，问题是视觉性的："这两个布局哪个更好？" — 这必须看到才能判断

判断标准只有一个："用户通过看到它是否能比读到它更好地理解？"如果是 → 浏览器。如果不是 → 终端。

06

终端状态锁定与技能链交接

▼
图 4：brainstorming → writing-plans 交接契约

6.1 终端状态锁定

**The terminal state is invoking writing-plans.** Do NOT invoke
frontend-design, mcp-builder, or any other implementation skill.
The ONLY skill you invoke after brainstorming is writing-plans.

为什么只允许 writing-plans：

  • 单一交接点 = 单一责任：writing-plans 是 brainstorming 到实现之间的唯一桥梁。它负责将 spec 转化为可执行的计划。如果允许直接调用 frontend-design 或 mcp-builder，就绕过了计划阶段，直接跳到实现。
  • 计划先行：writing-plans 的 description 明确说 "Use when you have a spec or requirements for a multi-step task, before touching code"。它强制在写代码之前先有计划。
  • 零上下文传递：writing-plans 假设执行计划的工程师"对代码库零上下文且品味可疑"。这个假设确保计划必须足够详细，不依赖任何隐含知识。

6.2 Spec 文档作为交接契约

writing-plans 读取 spec 中的以下内容来生成计划：

writing-plans 需要spec 中应该有
Goal（一句话目标）设计概述中的核心目标
Architecture（架构描述）架构部分的技术方案
Components（组件列表）组件部分的模块划分
Data flow（数据流）数据流部分的信息流向
Error handling（错误处理）错误处理部分的策略
Testing strategy（测试策略）测试部分的方案
Constraints（约束条件）讨论中确认的约束和限制

writing-plans 的 Self-Review 第一步就是 "Spec coverage"：

1. Spec coverage: Skim each section/requirement in the spec.
   Can you point to a task that implements it? List any gaps.

这直接验证了 spec → plan 的完整性。如果 spec 中有一个需求在 plan 中找不到对应任务，说明计划有遗漏。

6.3 零上下文原则的体现

Write comprehensive implementation plans assuming the engineer
has zero context for our codebase and questionable taste.
Document everything they need to know.

writing-plans 的 Overview 第一句就设定了零上下文假设。这意味着 spec 文档必须足够详细，使 writing-plans 能生成不依赖任何隐含知识的计划。这是 brainstorming 强制"写文档"而非"口头确认"的根本原因 — 口头讨论中的隐含知识无法传递给下游的子代理。

07

说服原则与防御层分析

▼

7.1 Authority（权威）

应用点源码中文
<HARD-GATE>Do NOT invoke any implementation skill, write any code, scaffold any project, or take any implementation action until you have presented a design and the user has approved it.在展示设计并获得用户批准之前，不得调用任何实现技能、写任何代码、脚手架化任何项目或采取任何实现行动。

7.2 Commitment（承诺）

- One question at a time - Don't overwhelm with multiple questions
- Multiple choice preferred - Easier to answer than open-ended

"一次一个问题" + "多选题优先" = 降低用户回答成本 → 提高参与度 → 增加对话深度 → 更多信息交换 → 更好的设计。这是 Commitment 原则的间接应用：不是直接要求用户承诺，而是通过降低承诺成本来增加承诺概率。

7.3 Scarcity（稀缺性）

- Incremental validation - Present design, get approval before
  moving on

"get approval before moving on" — 时序约束。不允许先进入下一步再回头获取批准。这创建了明确的"不能前进直到..."的门控。

7.4 Unity（统一/协作）

Help turn ideas into fully formed designs and specs through
natural collaborative dialogue.

"collaborative dialogue"（协作对话）= 我们一起工作。"natural"（自然）= 不是审讯，不是清单，而是像同事之间讨论问题一样。这是 Unity 原则的体现 — 将 agent 和用户定位为平等的协作者。

模式 1：双重门控模式

定义：对同一个产出物要求两次独立的用户确认，分别在"口头/草稿"和"书面/定稿"阶段。

适用场景：产出物将从"内部讨论"传递给"外部执行者"时。口头讨论中的隐含知识不会自动传递，书面化过程中产生的"转化内容"需要再次验证。

源码示例：

"Present design sections" -> "User approves design?"
"User approves design?" -> "Write design doc" [label="yes"]
...
"User reviews spec?" -> "Invoke writing-plans" [label="approved"]
"User reviews spec?" -> "Write design doc" [label="changes requested"]

业务 Skill 应用：在"API 设计"技能中：先口头讨论 API 端点 → 用户批准 → 写 OpenAPI spec 文档 → 用户再次审阅 spec → 交给实现团队。

模式 2：渐进式收敛模式

定义：通过"一次一个问题"的方式，逐步将模糊需求收敛为精确规格。

适用场景：需求不明确、用户自己也不确定想要什么时。

源码示例：

- Ask questions one at a time to refine the idea
- Prefer multiple choice questions when possible
- Only one question per message
- Focus on understanding: purpose, constraints, success criteria

关键设计："一次一个问题"不是慢，而是可控。每个问题的答案都会影响后续问题的设计。如果一次问 5 个问题，你无法根据第 1 个答案调整第 3 个问题。

业务 Skill 应用：在"需求分析"技能中：不是一次性问"你需要什么功能、什么约束、什么性能要求、什么安全要求"，而是先问"这个系统的主要用户是谁？"，根据答案再问"他们最常用的操作是什么？"

模式 3：方案对比模式

定义：总是提出 2-3 个方案并附带权衡分析，而不是只提出一个"最优"方案。

适用场景：有多个可行方案的技术决策。

源码示例：

- Propose 2-3 different approaches with trade-offs
- Present options conversationally with your recommendation
  and reasoning
- Lead with your recommended option and explain why

业务 Skill 应用：在"技术选型"技能中：提出 2-3 个数据库方案（PostgreSQL vs MongoDB vs SQLite），各自列出适用场景和限制，给出推荐并解释理由。

模式 4：Anti-Pattern 声明模式

定义：用明确的"反面案例 + 为什么反面"来阻止最常见的错误行为。

适用场景：有已知的常见失败模式时。

源码示例：

## Anti-Pattern: "This Is Too Simple To Need A Design"

Every project goes through this process. A todo list, a
single-function utility, a config change — all of them.
"Simple" projects are where unexamined assumptions cause
the most wasted work. The design can be short (a few sentences
for truly simple projects), but you MUST present it and
get approval.

关键设计：Anti-Pattern 不是简单说"不要这样做"，而是承认这种想法的合理性（"确实看起来简单"），然后解释为什么这种合理性是错的（"简单项目恰恰是最容易出问题的"），最后给出低成本的合规方式（"设计可以很短"）。

业务 Skill 应用：在"代码审查"技能中：Anti-Pattern: "This is just a minor change" — 小改动也可能引入大 bug。审查不需要很深，但必须执行。

模式 5：自检清单模式

定义：产出完成后立即执行结构化的自检，发现问题立即修复，不需要外部审查。

适用场景：检查项是机械性的、不需要人类判断的场景。

源码示例：

1. Placeholder scan: Any "TBD", "TODO", incomplete sections?
2. Internal consistency: Do any sections contradict each other?
3. Scope check: Focused enough for a single plan?
4. Ambiguity check: Could any requirement be interpreted two ways?

Fix any issues inline. No need to re-review — just fix and move on.

"No need to re-review" 的设计智慧：自检的修复不会引入新的问题类别，因此不需要迭代。但如果发现问题太多（超过 3 个占位符、多处理矛盾），应该考虑重写整个 section 而非逐个修复。

业务 Skill 应用：在"API 文档"技能中：写完 API 文档后自检 — 所有端点都有示例吗？错误码完整吗？认证方式一致吗？

9.1 拿到业务需求时，先回答这 5 个问题

  1. 这个需求解决谁的问题？ — 不是"用户"，而是具体哪类用户、什么场景下、什么触发条件
  2. 成功的标准是什么？ — 不是"功能完成了"，而是"什么可观察的结果说明成功了"
  3. 什么约束是硬性的？ — 技术栈、性能要求、安全要求、合规要求、时间限制
  4. 什么假设还没验证？ — "我们用 PostgreSQL"是假设，"用户需要实时通知"是假设。列出所有假设。
  5. 最坏情况是什么？ — 如果核心假设错了，最坏的影响是什么？能否接受？

如果你能回答这 5 个问题，你的需求已经比大多数"直接动手"的情况清晰 10 倍。

9.2 如何用"一次一个问题"探索不确定的业务领域

假设你想为团队设计一个"代码审查自动化"工具，但不确定具体需求：

Q1
当前代码审查的最大痛点是什么？
如果回答"太慢" → 后续问"是哪个环节慢：创建 PR、分配审查者、实际审查、汇总反馈？"
Q2
审查者通常是同一个人还是不同的人？
如果回答"通常同一个人" → 关注"减少审查者疲劳"；如果回答"不同的人" → 关注"知识传递和标准一致性"。
Q3
审查中发现的常见问题类型是什么？
这个答案直接影响工具的核心功能 — 如果主要是"风格问题"，工具需要 lint 集成；如果是"逻辑错误"，工具需要静态分析。

注意：每个问题的答案都改变了下一个问题的方向。这就是"一次一个问题"的力量。

9.3 如何判断自己是否"想清楚了"

你还没想清楚的信号：

  • 你用的词是"大概"、"应该"、"可能"、"差不多"
  • 你无法用一句话描述"这个东西做完后，用户会看到什么变化"
  • 你无法列出 3 个以上的具体文件/模块/接口名称
  • 你无法说出"如果 X 出错了，应该怎么办"
  • 你还没考虑过"什么情况下不应该做这个功能"

你想清楚的信号：

  • 你可以说出"这个功能的目标是 [一句话]，成功的标志是 [可观察结果]"
  • 你可以画出组件之间的数据流向
  • 你可以列出所有已确认的约束和所有未验证的假设
  • 你可以回答"如果去掉这个功能，用户还能用吗？如果不能，为什么？"

9.4 从模糊想法到精确 Skill 需求的转化路径

假设你有一个模糊的想法："我想让 AI 更好地帮我写代码"。以下是转化路径：

1
问题诊断（不是功能定义）
不是"我要一个代码生成技能"，而是"AI 写的代码有什么具体问题？是风格不一致？是缺少测试？是不遵循项目约定？"
2
约束收集
不是"功能要有什么"，而是"在什么条件下使用？什么触发？什么场景不适用？与现有工具的冲突是什么？"
3
方案对比
A) 修改 CLAUDE.md 添加规则？B) 创建一个 discipline-enforcing skill？C) 修改现有 skill？各自的 trade-off 是什么？
4
假设验证
用 brainstorming 的基线测试方法：在没有这个 skill 的情况下，AI 会犯什么错？记录具体的失败模式。这些就是你需要在 skill 中解决的问题。
5
规格文档化
将以上所有内容写成 spec 文档：问题定义、触发条件、成功标准、约束、假设、方案选择及理由。让另一个人（或另一个 agent）审阅。

完成这 5 步后，你就有了 brainstorming 技能所要求的"fully formed design"。从"模糊想法"到"精确需求"的转化，不是灵感，而是结构化的思考流程。
