01

writing-plans 在 Agentic SDLC 中的位置

▼

1.1 技能链全景

writing-plans 是 Superpowers Agentic SDLC（软件开发生命周期）中的第二个核心技能。它处于 brainstorming（规格产出）和 subagent-driven-development / executing-plans（代码实现）之间，承担"承上启下"的关键角色。

Brainstorming 的终端状态是调用 writing-plans。 "The terminal state is invoking writing-plans." — brainstorming 技能明确禁止调用任何其他实现类技能。

这条技能链的设计遵循一个清晰的"信息递减"原则：每一步都在将隐含知识转化为显式文档，确保下一个环节不需要任何隐含上下文就能工作。

图 1：Agentic SDLC 全链路
docs/superpowers/specs/) Contract: 通过用户审批门的已批准规格 terminal state writing-plans 承上启下 · 最高信息损耗交接点 Input: spec 文档 + 项目上下文（worktree 环境） Process: Scope Check → File Structure → Task 分解 → No Placeholders → Self-Review Output: plan 文档 (docs/superpowers/plans/) Contract: 零上下文可执行计划 — 任何 agent 无需额外信息即可执行 recommended alternative subagent-driven-development Input: plan 文档 Process: 每个 Task 派发独立 subagent + 双阶段审查 Output: 代码 + 测试 + commits executing-plans Input: plan 文档 Process: 批量执行 + checkpoint 审查 Output: 代码 + 测试 + commits

1.2 为什么这是"最高信息损耗交接点"

在整个 Agentic SDLC 中，brainstorming → writing-plans → executing agent 的交接链里，writing-plans 是信息损耗最大的环节。原因如下：

信息损耗分析：

  • Brainstorming → writing-plans：Planning agent 拥有完整的项目上下文（代码库、设计决策、用户意图）。它能"理解"隐含假设。但 spec 文档只能记录显式信息。
  • writing-plans → executing agent：执行 agent 是一个全新的进程，没有 brainstorming 的对话历史，没有 design 的迭代过程，没有"为什么"的上下文。它只有 plan 文档里的文字。
  • 损耗发生在哪里：Planning agent 在写 plan 时"觉得不用说"的信息（隐含的设计意图、命名惯例、已知约束），执行 agent 完全不知道。

这就是为什么 writing-plans 强制执行 "zero context" 原则 — assuming the engineer has zero context for our codebase and questionable taste。这不是夸张，这是对信息损耗的现实评估。

1.3 上游约束与下游期望

维度上游（brainstorming）writing-plans下游（subagent-driven / executing）
输入产物spec 文档（已审批）spec + 项目上下文plan 文档
输出产物spec 文档plan 文档代码 + 测试 + commits
质量门控用户审批门（2 道）Self-Review（3 项检查）双阶段审查 / checkpoint
上下文需求对话历史 + 项目文件spec + worktree 环境plan 文档（零额外上下文）
失败模式设计未审批就动手plan 有 placeholderagent 遇到歧义自行猜测

关键洞察：writing-plans 的设计目标是让下游不需要"猜测"。每一个决策（文件路径、函数签名、测试用例、预期输出）都必须在 plan 中显式记录。这不是"详细"的问题，这是"信息完整性"的问题。

02

Plan Document 结构分析

▼

2.1 整体层次结构

Plan 文档采用三层嵌套结构：Header（计划元信息）→ Task（任务单元）→ Step（执行步骤）。每一层解决不同的问题，缺失任何一层都会导致下游执行失败。

图 2：Plan 文档的层次结构

2.2 Header 字段深度分析

REQUIRED SUB-SKILL 声明

Plan 文档的第一行是一个关键的设计决策 — 它不是给人类看的注释，而是给执行 agent 看的指令：

> **For agentic workers:** REQUIRED SUB-SKILL: Use
superpowers:subagent-driven-development (recommended)
or superpowers:executing-plans to implement this plan
task-by-task. Steps use checkbox (- [ ]) syntax for tracking.

// 对于 agentic 执行者：必须使用指定的子技能来实现此计划。
// 步骤使用 checkbox 语法进行跟踪。

这个设计解决了一个实际问题：执行 agent（subagent）不知道自己应该用什么方法论来执行 plan。通过在 plan 文档本身中嵌入执行指令，plan 文档变成了一个"自包含的执行合约"。

Goal

**Goal:** [One sentence describing what this builds]
目标：[一句话描述这个计划构建什么]

为什么是一句话：如果需要超过一句话才能描述目标，说明这个 plan 的范围可能过大，应该拆分成多个 plan。这呼应了 Scope Check 的设计。

Architecture

**Architecture:** [2-3 sentences about approach]
架构：[2-3 句话描述方法]

为什么限制 2-3 句：Architecture 不是设计文档（那是 brainstorming 的产物），而是给执行 agent 的决策框架。执行 agent 需要知道"大方向"以便在遇到边界情况时做出正确的判断，但不需要（也不应该）知道所有的设计细节。

Tech Stack

**Tech Stack:** [Key technologies/libraries]
技术栈：[关键技术/库]

为什么列出技术栈：执行 agent 可能在 import 时不知道该用哪个库。例如，项目使用 pytest 还是 unittest，使用 requests 还是 httpx — 这些信息对正确实现至关重要。

2.3 Task 的 Files 列表

**Files:**
- Create: `exact/path/to/file.py`
- Modify: `exact/path/to/existing.py:123-145`
- Test: `tests/exact/path/to/test.py`

文件列表：
- 创建：精确文件路径
- 修改：精确文件路径 + 行号范围
- 测试：精确测试文件路径

为什么必须包含行号：Modify: existing.py:123-145 告诉执行 agent 精确的修改位置。没有行号，执行 agent 需要自己找到正确的修改位置 — 这对"零上下文"的 agent 来说是一个不必要的障碍和潜在的出错点。

03

Task 结构深度拆解

▼

3.1 TDD 五步循环

每个 Task 被强制分解为 5 个步骤，对应 TDD 的 Red-Green-Commit 微循环。这不是"建议"，而是强制结构 — 每个 Task 必须有且仅有这 5 个步骤。

图 3：单个 Task 的 TDD 五步循环

3.2 为什么每个 Step 只做一件事

五步结构的每一步都是单一原子操作。这不是随意的设计，而是基于以下观察：

  • 认知负荷最小化：执行 agent 的上下文窗口有限。每步只做一件事，减少 agent 在单步中出错的可能。
  • 失败隔离：如果某一步失败，可以精确定位问题。Step 1 失败 = 测试设计有问题；Step 2 失败 = 测试逻辑有误；Step 3 失败 = 实现有 bug；Step 4 失败 = 实现不完整。
  • 进度可追踪：5 步对应 5 个 checkbox，执行进度一目了然。
  • 2-5 分钟粒度："Each step is one action (2-5 minutes)" — 这个时间窗口确保每步足够小以至于不容易出错，但又足够大以至于有实质产出。

3.3 为什么 Verify RED 和 Verify GREEN 不能跳过

这是 writing-plans 中最容易被"优化掉"的两个步骤。技能通过 TDD 子技能的铁律来阻止跳过：

NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
没有失败测试就没有生产代码

具体原因：

跳过 Verify RED

测试可能通过了 — 但不是因为实现正确，而是因为测试本身在测试已有行为（而非期望行为）。你没有证明"测试能捕获 bug"，你只是证明"测试能通过"。

TDD 子技能的原话："If you didn't watch the test fail, you don't know if it tests the right thing."

跳过 Verify GREEN

实现可能没有让测试通过 — 引入了新的 bug。如果没有验证，你不知道自己写的东西是否真的工作了。回归可能在你到达下一个 Task 时才被发现，那时候调试成本已经增加了。

3.4 为什么代码块必须是完整代码，不是伪代码

Steps that describe what to do without showing how
(code blocks required for code steps) — plan failure
只描述做什么但不展示怎么做的步骤——计划失败

执行 agent 是"zero context"的。如果你写 "Add a function that validates email"，执行 agent 需要自己决定：

  • 函数叫什么名字？
  • 参数类型是什么？
  • 返回值是什么？
  • 验证规则是什么？
  • 错误时抛异常还是返回 None？

每一个"自己决定"都是一个潜在的偏差点。plan 的目标就是消除这些偏差点。

3.5 为什么命令必须包含预期输出

- [ ] **Step 2: Run test to verify it fails**

Run: `pytest tests/path/test.py::test_name -v`
Expected: FAIL with "function not defined"

运行：精确命令
预期：失败并显示 "function not defined"

"Expected output" 不是一个可选的装饰 — 它是执行 agent 的验证标准。没有预期输出，执行 agent 不知道：

  • 失败是否是"正确的失败"（fail for the right reason）
  • 通过是否是"完全通过"（all tests passing, no warnings）
  • 输出中是否有不应该出现的 warning 或 error
04

No Placeholders — 零上下文交接原则

▼

4.1 核心原则

writing-plans 中最具防御性的设计就是 No Placeholders 规则。技能明确列出了 6 种"plan failures" — 不是"bad practices"，而是失败：

Every step must contain the actual content an engineer needs.
These are **plan failures** — never write them:
每个步骤必须包含工程师需要的实际内容。
这些是**计划失败**——永远不要写它们：
图 4：零上下文交接 vs 有上下文交接

4.2 六种 Plan Failures 详解

类型错误示例为什么是失败
TBD / TODO "TBD: implement later" 执行 agent 不知道 "later" 是什么时候，也不知道要实现什么。Plan 中不应有未完成项。
模糊指令 "Add appropriate error handling" "appropriate" 对 planning agent 和 executing agent 可能意味着完全不同的东西。
空测试引用 "Write tests for the above" 没有实际测试代码，执行 agent 不知道要测什么、怎么测、期望什么。
交叉引用 "Similar to Task 3" 执行 agent 可能跳着读 Task，或在不同 session 中执行。每个 Task 必须自包含。
描述代替代码 "Create a function to parse dates" 描述了"做什么"但没有"怎么做"。函数签名、参数类型、返回值 — 全部缺失。
幽灵引用 调用未在任何 Task 中定义的类型/函数 执行 agent 无法找到引用的定义，导致编译失败或实现错误。

4.3 为什么 "Similar to Task N" 被禁止

"Similar to Task N" (repeat the code — the engineer may
be reading tasks out of order)
"类似于 Task N"（重复代码——工程师可能不按顺序阅读任务）

这个禁令背后的原因比表面看到的更深：

  • Subagent 场景：subagent-driven-development 为每个 Task 派发一个全新的 subagent。Task 7 的 subagent 无法访问 Task 3 的内容。
  • 跳读场景：执行 agent 可能因为之前的 Task 失败而跳到某个 Task 重新执行。
  • Session 切换场景：executing-plans 可能在不同的 session 中执行。不同 session 之间不共享上下文。

设计原则："every handoff receiver is a brand new agent"（每个交接接收者都是全新的 agent）。这不是假设，这是实际运行时的现实。

4.4 为什么 "Add appropriate error handling" 不够

"appropriate" 是一个主观判断词。对于 planning agent 来说，"appropriate error handling" 可能意味着：

  • 抛出特定类型的异常（如 ValidationError）
  • 记录日志并返回默认值
  • 向上传播错误让调用者处理
  • 显示用户友好的错误消息

对于执行 agent 来说，这四种选择产生的代码完全不同。如果 plan 不明确指定，执行 agent 会基于自己的"判断"来选择 — 这个判断可能与设计意图完全不同。

05

File Structure 与 Task 分解的设计决策

▼

5.1 从 Spec 到 File Structure 到 Task 的分解路径

writing-plans 的分解过程不是一步到位的。它遵循一个清晰的三层分解路径：Spec → File Structure → Tasks。每一层都在上一层的基础上进一步细化。

图 5：从 Spec 到 File Structure 到 Task 的分解路径

5.2 "Files that change together should live together"

Files that change together should live together.
Split by responsibility, not by technical layer.
一起变更的文件应该放在一起。
按职责拆分，而不是按技术层拆分。

这是一个反直觉的设计决策。传统的分层架构（controllers、services、models、views）将文件按技术角色组织。但 writing-plans 推荐的是按职责组织：

按技术层拆分
src/
  models/user.py
  services/auth.py
  controllers/login.py
  views/login_view.py

改一个登录功能要改 4 个目录下的文件。

按职责拆分
src/auth/
  user.py
  service.py
  login.py
  login_view.py

改登录功能只涉及 src/auth/ 目录。

为什么这对 plan 至关重要：Task 的粒度与文件的职责范围直接相关。如果文件按技术层拆分，一个 Task 可能需要跨多个目录修改文件，增加了执行复杂度和出错概率。按职责拆分后，每个 Task 更可能只涉及一个目录下的文件。

5.3 Scope Check — 何时拆分为多个 Plan

If the spec covers multiple independent subsystems, it should
have been broken into sub-project specs during brainstorming.
If it wasn't, suggest breaking this into separate plans — one
per subsystem. Each plan should produce working, testable
software on its own.
如果 spec 涵盖多个独立子系统，应该被拆分为子项目规格。
每个 plan 应该产出独立的、可测试的软件。

Scope Check 的设计意图是防止 plan 过大导致执行失败。一个包含 20+ Tasks 的 plan 比 4 个各包含 5 Tasks 的 plan 更容易出问题：

  • 上下文溢出：Planning agent 在写后面的 Task 时可能已经忘了前面的设计决策
  • Self-Review 效果递减：检查 20 个 Task 的一致性比检查 5 个 Task 困难得多
  • 执行中断风险：如果 Task 15 失败，可能需要重新审视整个 plan
  • 并行性丧失：大 plan 无法并行执行（虽然 subagent-driven 目前是串行的，但结构上应该允许未来并行）

5.4 为什么 "2-5 分钟" 粒度

**Each step is one action (2-5 minutes):**
- "Write the failing test" - step
- "Run it to make sure it fails" - step
- "Implement the minimal code to make the test pass" - step
- "Run the tests and make sure they pass" - step
- "Commit" - step
每一步是一个动作（2-5 分钟）

"2-5 minutes" 不是随意选择的时间窗口。它与 LLM 的行为特征直接相关：

  • 太短（<2 min）：步骤过于琐碎（如"打开文件"），增加 plan 长度但不增加价值
  • 太长（>5 min）：步骤过于复杂，执行 agent 在单步中出错概率上升，失败时难以定位
  • 2-5 min：恰好对应"写一个测试函数"或"实现一个小函数"的时间 — 这是一个 agent 可以可靠完成的原子操作
06

Self-Review 机制

▼

6.1 三项检查流程

Self-Review 是 writing-plans 的内置质量保证机制。它在 plan 写完之后、交付执行之前运行。值得注意的是，这是一个自检机制，不是外部审查。

图 6：Self-Review 三项检查流程

6.2 为什么是自检而不是外部审查

After writing the complete plan, look at the spec with fresh
eyes and check the plan against it. This is a checklist you
run yourself — not a subagent dispatch.
写完完整计划后，用全新的眼光审视 spec 并对照检查。
这是你自己运行的清单——不是 subagent 调度。

这个设计选择有几个原因：

  • 速度：自检比派发 subagent 更快。Planning agent 已经有了完整的上下文，不需要重新加载 spec 和 plan。
  • 即时修复：发现问题后可以立即修复 — "Fix inline. No need to re-review"（内联修复，无需重新审查）。
  • 外部审查仍然存在：plan-document-reviewer-prompt.md 提供了可选的外部审查。Self-Review 是第一道防线，外部审查是第二道。

6.3 Type Consistency — clearLayers() vs clearFullLayers() 的 Bug

Self-Review 中最精妙的检查是 Type Consistency，因为它捕获的是跨 Task 的一致性错误 — 这类错误在编写 plan 时极难发现，但在执行时会导致编译失败。

技能原文的例子：

A function called `clearLayers()` in Task 3 but
`clearFullLayers()` in Task 7 is a bug.
Task 3 中叫 `clearLayers()` 但 Task 7 中叫 `clearFullLayers()`
就是一个 bug。

这类 bug 的产生机制：

  1. Task 3 定义了一个函数 clearLayers()
  2. 写到 Task 7 时，planning agent "记错了"函数名，写成了 clearFullLayers()
  3. 执行 agent 在 Task 7 调用 clearFullLayers() → 编译/运行失败
  4. 执行 agent 不知道 Task 3 中定义的是 clearLayers()（因为它是"零上下文"的）
  5. 执行 agent 卡住 → 需要 human 介入

6.4 "Fix inline. No need to re-review"

If you find issues, fix them inline. No need to re-review
— just fix and move on. If you find a spec requirement
with no task, add the task.
如果发现问题，内联修复。无需重新审查——直接修复并继续。
如果发现 spec 需求没有对应的 Task，添加 Task。

这条规则防止了一个常见的反模式：过度审查循环。如果每次修复后都需要重新审查，planning agent 可能会陷入"修复 → 审查 → 发现新问题 → 修复 → 审查"的无限循环。技能明确说"修复后继续前进"，这是对效率的合理权衡。

6.5 外部审查 — plan-document-reviewer-prompt.md

除了自检，writing-plans 还提供了可选的外部审查模板：

审查类别检查内容
CompletenessTODOs、placeholders、不完整的 tasks、缺失的步骤
Spec AlignmentPlan 覆盖 spec 需求，无重大范围蔓延
Task DecompositionTasks 有清晰边界，步骤可执行
Buildability工程师能否在不卡住的情况下遵循此 plan？

外部审查有一个关键的校准原则：

"Only flag issues that would cause real problems during implementation." 只标记会在实现过程中造成实际问题的内容。一个工程师构建了错误的东西或卡住了，这是问题。措辞、风格偏好和"锦上添花"的建议不是问题。
07

Execution Handoff — 双模式执行选择

▼

7.1 两种执行模式

Plan 写完后，writing-plans 提供两种执行模式供用户选择。这不是"随便选一个" — 每种模式有不同的适用场景和权衡。

图 7：两种执行模式的对比

7.2 为什么 Subagent-Driven 是推荐模式

writing-plans 明确将 subagent-driven-development 标记为 "(recommended)"。原因与 writing-plans 的核心设计原则直接相关：

核心一致性：writing-plans 的设计假设执行者是"零上下文"的。subagent-driven-development 正好满足这个假设 — 每个 Task 的 subagent 确实没有任何前置上下文。而 executing-plans 在同一 session 中执行，理论上存在上下文泄露的可能。

subagent-driven-development 的核心原则也呼应了这一点：

"They should never inherit your session's context or history — you construct exactly what they need." 它们永远不应该继承你的 session 上下文或历史 — 你精确构建它们需要的内容。

7.3 两种模式的共同前置要求

无论选择哪种执行模式，都必须先使用 using-git-worktrees 创建隔离工作区：

// 两种模式的共同前置要求：
1. using-git-worktrees — REQUIRED
   - 创建隔离工作区
   - 验证 .gitignore
   - 运行基线测试
   - 报告就绪状态

2. Review plan critically first
   - 识别问题或疑虑
   - 有问题 → 先向 human 提出

using-git-worktrees 的 REQUIRED 标记确保了执行环境的安全隔离。即使执行过程中出错，也不会影响主分支或其他工作。

7.4 executing-plans 的关键约束

executing-plans 虽然是备选方案，但仍然有严格的质量约束：

## When to Stop and Ask for Help

**STOP executing immediately when:**
- Hit a blocker (missing dependency, test fails, instruction unclear)
- Plan has critical gaps preventing starting
- You don't understand an instruction
- Verification fails repeatedly

何时停止并寻求帮助：
遇到以下情况立即停止执行：
- 遇到阻碍（缺失依赖、测试失败、指令不清）
- Plan 有阻止启动的关键缺口
- 你不理解某条指令
- 验证反复失败

这条规则的本质是：宁可停下来求助，也不要猜测着继续。猜测着继续产生的错误代码比停下来求助的成本高得多。

08

说服原则与防御层分析

▼

8.1 四层说服原则

writing-plans 的文本中嵌入了多个说服心理学原则。这些不是偶然的措辞选择 — 它们是精心设计的行为干预，旨在阻止 agent 跳过关键步骤。

Authority（权威）

技能通过绝对化的语言建立权威感：

"NEVER write them"        — 永远不要写它们
"plan failures"             — 计划失败
"MANDATORY. Never skip."   — 强制的。永远不要跳过。

"plan failures" 这个措辞将 No Placeholders 的违规行为从"bad practice"提升到了"failure"级别。这不是"建议不要做"，而是"做了就是失败"。

Commitment（承诺）

**Announce at start:** "I'm using the writing-plans skill
to create the implementation plan."
开始时宣布："我正在使用 writing-plans 技能来创建实现计划。"

公开宣布使用特定技能，等于做了一个公开承诺。一旦宣布，agent 在后续步骤中跳过该技能的要求会产生认知不一致，从而增加遵守的概率。

Social Proof（社会证明）

"Every step"                  — 每一步
"DRY. YAGNI. TDD. Frequent commits."
"assuming the engineer has zero context"

"Every step"、"DRY. YAGNI. TDD. Frequent commits." 这些是行业最佳实践的浓缩表达。它们暗示"这是所有优秀工程师都遵循的方法"，利用社会证明来增强说服力。

8.2 四层防御架构

writing-plans 的防御不是单一层面的。它构建了一个四层纵深防御体系，每一层都独立阻止一种特定的跳过行为。

防御层机制阻止的行为违规后果
L1: 权威铁律 "NEVER write them"、"plan failures"、绝对化语言 整体性跳过（如忽略 No Placeholders 规则） 技能层面的明确禁止，agent 无法"合理化"跳过
L2: 反合理化 No Placeholders 的 6 种具体模式 + 反例 "这个 placeholder 可以接受"的自我合理化 每种合理化路径都被预判和封堵
L3: 序列门控 5 步固定结构 + checkbox 语法 跳过中间步骤（如跳过 Verify RED） 结构不允许跳步 — 5 步是固定序列
L4: 证据验证 Verify RED + Verify GREEN + 预期输出 执行了步骤但没有真正验证 必须有可观察的证据（输出匹配预期）

防御层的协同效应：这四层防御不是简单堆叠 — 它们覆盖了不同的"攻击面"：

  • L1 阻止"我不需要遵守这个规则"的念头
  • L2 阻止"这个特殊情况可以例外"的合理化
  • L3 阻止"我可以跳过这一步"的操作
  • L4 阻止"我做了但没做对"的表面合规

即使 agent 成功绕过一层（例如忽略 L1 的权威语言），后续的防御层仍然会捕获违规行为。

8.3 TDD 铁律的防御角色

writing-plans 本身不定义 TDD 规则 — 它引用 test-driven-development 子技能的铁律。这个设计选择有两层含义：

  1. 权威转移：writing-plans 不是在"自己说"TDD 很重要，而是在引用一个独立的、专门定义 TDD 规则的技能。这增加了说服力 — "这不是我说的，这是 TDD 技能说的"。
  2. 执行时生效：铁律真正生效的时机不是写 plan 时，而是执行 plan 时。subagent-driven-development 要求 subagent 使用 TDD 技能。
09

可复用的设计模式提取

▼

writing-plans 中包含多个可推广的设计模式。这些模式不局限于 plan 编写 — 它们适用于任何需要"将隐含知识转化为显式指令"的场景。

9.1 零上下文交接模式

定义：假设信息接收者没有任何前置上下文，所有必要信息必须显式编码在交付物中。

适用场景：任何涉及 agent 间交接、跨 session 传递信息、或人类与 AI 协作的场景。

源码示例：

Write comprehensive implementation plans assuming the engineer
has zero context for our codebase and questionable taste.
Document everything they need to know: which files to touch
for each task, code, testing, docs they might need to check,
how to test it.
编写全面的实现计划，假设工程师对我们的代码库零上下文
且品味可疑。记录他们需要知道的一切。

业务技能应用：当撰写 API 文档时，假设调用者没有看到任何内部设计文档。每个端点的请求格式、响应格式、错误码、认证方式都必须自包含。

9.2 No Placeholders 模式

定义：显式列出所有不允许的模糊表达模式，为每种模式提供具体示例和替代方案。

适用场景：任何需要确保指令完整性的场景 — SOP 编写、操作手册、agent prompt 设计。

源码示例：

These are **plan failures** — never write them:
- "TBD", "TODO", "implement later", "fill in details"
- "Add appropriate error handling" / "add validation"
- "Similar to Task N" (repeat the code)
- Steps that describe what to do without showing how
- References to types, functions, or methods not defined
  in any task
这些是**计划失败**——永远不要写它们

业务技能应用：当编写部署 checklist 时，不要写"配置适当的环境变量"，而是写"设置 DATABASE_URL=postgres://..."。不要写"确保安全设置正确"，而是列出每一条安全规则。

9.3 TDD 微循环模式

定义：将每个工作单元强制分解为"写测试 → 验证失败 → 写实现 → 验证通过 → 提交"的固定 5 步循环。

适用场景：任何需要确保变更正确性的场景 — 代码修改、配置变更、数据迁移。

源码示例：

**Each step is one action (2-5 minutes):**
- "Write the failing test" - step
- "Run it to make sure it fails" - step
- "Implement the minimal code to make the test pass" - step
- "Run the tests and make sure they pass" - step
- "Commit" - step
每一步是一个动作（2-5 分钟）

业务技能应用：当修改数据库 schema 时：(1) 写一个验证当前 schema 的测试并确认通过 (2) 写一个验证新 schema 的测试并确认失败 (3) 执行 migration (4) 确认新测试通过 (5) 提交。即使没有"代码"，同样的模式确保了变更的正确性。

9.4 文件先行模式

定义：在定义具体执行步骤之前，先确定所有涉及的文件及其职责。文件结构决定任务分解。

适用场景：任何需要多文件协作的项目 — 代码重构、新功能开发、文档编写。

源码示例：

Before defining tasks, map out which files will be created
or modified and what each one is responsible for. This is
where decomposition decisions get locked in.
在定义任务之前，先规划将创建或修改哪些文件，
以及每个文件负责什么。这是分解决策被锁定的地方。

业务技能应用：当规划一个大型文档重构时，先列出所有文档文件、每个文件的职责范围、文件之间的引用关系，然后再逐步编写每个文件。

9.5 自检清单模式

定义：在交付之前运行一个固定的自检清单，发现问题立即修复，不需要外部审查循环。

适用场景：任何需要质量保证但又不值得派发独立审查的场景 — 快速迭代、个人项目、内部工具。

源码示例：

**1. Spec coverage:** Can you point to a task that implements it?
**2. Placeholder scan:** Search for red flag patterns.
**3. Type consistency:** Do names match across tasks?

If you find issues, fix them inline. No need to re-review
— just fix and move on.
如果发现问题，内联修复。无需重新审查。

业务技能应用：当提交 PR 之前，运行自检清单：(1) 所有文件都有合理的 commit message？(2) 没有 console.log / debug 代码残留？(3) 新增的 API 都有文档？发现问题立即修复，不需要反复审查。

模式之间的关系：这五个模式不是孤立的 — 它们构成了一个完整的"知识传递管道"：

  1. 文件先行模式决定结构 → "改哪些文件"
  2. TDD 微循环模式决定步骤 → "每步做什么"
  3. No Placeholders 模式决定内容 → "每步写什么"
  4. 零上下文交接模式决定标准 → "写到什么程度"
  5. 自检清单模式决定质量 → "交付前检查什么"
