# 个人需求与背景说明

## 我是谁

我是一名低代码/零代码报表平台的从业者。我们的平台通过拖拉拽组件的方式让用户搭建报表页面。组件可以配置数据源，数据源关联到数据集，数据集包含宽表、明细表、维度表等配置。平台的渲染流程是：用户配置 → 元数据存储 → 预览时通过 chart/data 等固定接口请求数据 → 组件接收数据渲染。

## 我的当前水平

- 我已经理解了 superpowers 开源项目的核心架构设计（五层防御体系、说服三原则、反合理化表格、零上下文交接等概念）
- 我能使用 skill-creator 等 AI 工具生成 skill，也能审查 skill 的流程是否符合预期
- 我对 SQL、数据库表结构（宽表/明细表/维度表）、索引等概念有基本理解，但不是专家
- 我的核心短板：**在动手写 skill 之前，没有把业务问题想清楚的习惯。** 往往是模糊的想法直接丢给 AI 生成 skill，结果是"能用但不好用"，解决不了实际的业务问题

## 我的核心目标

**提升三个能力：**

1. **识别能力**：看到别人的 skill，能判断它写得好不好、为什么好或不好
2. **设计能力**：遇到业务问题时，能先通过结构化思考（类似 brainstorming 的方式）搞清楚问题和需求，再设计 skill 的方案
3. **写作能力**：能写出高质量的、解决实际业务问题的 skill，包含具体的约束规则、反合理化防护、When/Then 触发器等设计模式

## 我当前的具体业务需求

我需要编写一个**排查报表页面性能的 skill**。以下是完整需求：

### 业务背景

- 平台：拖拉拽组件生成报表的低代码平台
- 问题：客户反馈某些报表页面加载慢，需要一套标准化的排查流程
- 目标：输入一个线上报表 URL，自动采集性能数据，分析瓶颈，输出具体的优化建议报告

### 性能数据的四个采集层

**Layer 0 — 常规 Web 性能基线**（Playwright 采集）
- Core Web Vitals：TTFB、FCP、LCP、CLS
- 资源加载：JS/CSS/图片/字节数量与体积
- 网络请求瀑布图：请求总数、并发、阻塞
- DOM 复杂度：节点数量

**Layer 1 — 业务接口性能**（Playwright 拦截网络请求 + React Fiber 获取组件实例信息）
- 元数据接口响应时间
- 每个组件的 chart/data 接口：URL、响应时间、返回数据量
- 重复请求检测、串行阻塞检测

**Layer 2 — 数据层性能**（获取 SQL + 分析）
- 每个组件生成的 SQL 查询
- SQL 的 WHERE/GROUP BY/JOIN 分析（是否有索引、是否全表扫描）
- 数据集配置（宽表/明细表/维度表）是否合理

**Layer 3 — 渲染性能**（Playwright Performance API）
- 每个组件渲染耗时
- 组件间渲染是否串行阻塞

### 报告输出要求

报告分两部分，**关键是必须具体，不能泛化**：

**第一部分：给搭建者（非开发人员）的优化建议**
- 必须包含具体的组件名称、数据集名称、字段名称
- 例如：不能说"合并组件"，要说"把「销售趋势图」和「月度对比图」合并"
- 例如：不能说"改表配置"，要说"将数据集从「订单宽表」切换为「订单明细表」"
- 对于平台层面的问题（如重复请求、缺少索引），建议搭建者提工单

**第二部分：给开发人员的优化方向**
- 关联到源码仓库的具体模块位置
- 例如："模块 X 中组件 A 和 B 分别请求了相同接口，可提升到父组件统一请求"

### 执行环境

- 在源码仓库中通过 Claude Code 执行
- 可用 Playwright 打开浏览器访问目标 URL
- 可通过 React Fiber 获取页面上组件的实例和元数据
- 可读取源码仓库中的代码
- 已有 skill 可获取组件元数据和页面元数据
