Superpowers 深度设计伴读

这个学习目录不是为了“快速使用 Superpowers”，而是为了拆解它为什么能约束 AI 编程代理、如何把技能串成方法论，以及哪些设计可以迁移到你自己的项目。

学习目标

从 README 理解整体方法论，从 skills 理解行为塑形，从 CLAUDE.md / AGENTS.md 理解质量门禁，从 tests / docs 理解它如何证明这些流程真的有效。

README先建立整体工作流和哲学框架。 CLAUDE.md看它如何约束 Claude Code 贡献行为。 AGENTS.md看跨 agent 的共同贡献守则。 Bootstrap Skill理解自动触发体系从哪里开始。

遇到问题，去哪里看？

下面不是文件目录复刻，而是按学习问题组织的入口。每张卡都指向真实仓库文件。

workflow

我想知道它到底解决什么问题

看 Basic Workflow 和 Philosophy：Superpowers 把 AI 编程从自由发挥改造成分阶段、可审查、可验证的方法论。

 design

为什么写代码前要先设计？

看 brainstorming skill：它用强门禁阻止 agent 直接开写，先把模糊想法转成可确认的设计。

 planning

设计通过后如何变成可执行任务？

看 writing-plans：它要求计划具体到文件、步骤、验证方式，降低后续 agent 自由解释空间。

 subagents

为什么要用子代理执行？

看 subagent-driven-development：每个任务隔离执行，再经过规格符合性和代码质量两轮审查。

 tdd

它怎样避免“看起来完成了”？

看 test-driven-development：红绿重构把“声明完成”换成“先失败、再通过”的证据链。

 debug

遇到 bug 为什么不能猜？

看 systematic-debugging：它要求先找根因，再做最小修复，避免 agent 用补丁掩盖问题。

 review

代码审查在流程中怎么卡住质量？

看 requesting-code-review：审查不是礼貌动作，而是继续前进前的质量门。

 evidence

它怎么测试“技能真的改变行为”？

看 docs/testing.md 和 tests：重点是运行真实 Claude Code 会话并验证 transcript，而不是只检查文本。

 harness

它如何适配不同编码工具？

看 OpenCode 文档和 GEMINI.md：同一套方法论通过 bootstrap、技能注册和工具映射适配不同 harness。

核心设计地图

Superpowers 的关键不是“有很多技能”，而是技能之间组成了一条有门禁、有证据、有复盘的开发流水线。

mainline

开发主链路

01用户意图进入后，先触发 using-superpowers / brainstorming，而不是直接实现。
02brainstorming 把想法变成设计，并要求用户确认。
03writing-plans 把设计拆成小任务和验证步骤。
04using-git-worktrees 隔离实现环境，降低污染主线风险。
05TDD、subagent-driven-development、code review 形成执行和审查闭环。
06finishing-a-development-branch 做完成前验证和收尾选择。
support

支撑系统

ACLAUDE.md / AGENTS.md 保护 human partner，不让 agent 产出低质量 PR。
Bskills 目录把行为协议模块化，每个技能负责一个可触发场景。
Chooks 和 harness 文档负责把方法论接入不同工具。
Dtests 用真实会话 transcript 验证 agent 是否真的按流程行动。
E贡献规则阻止 speculative、bulk、domain-specific、fabricated 这类高风险改动。

迁移到你自己的项目

学习 Superpowers 时，不要只复制文件；更值得迁移的是它把 agent 行为收窄到“可确认、可执行、可验证”的设计方式。

copy idea

行为塑形

用强约束、红旗表、不可跳过的检查点，降低 agent 自作主张。

 copy idea

流程链

把“做功能”拆成设计、计划、实现、审查、完成验证，而不是一次性提示词。

 copy idea

证据优先

要求测试、transcript、diff、真实问题描述，让完成状态有证据。

 do not copy blindly

不要照抄什么

不要直接照搬 Superpowers 的语气、PR 规则、跨 harness 复杂度；先适配你的项目规模。

 skill design

如何写自己的技能

看 writing-skills，理解技能不是普通文档，而是会塑造 agent 行为的操作协议。

 evaluation

如何验证技能有效

看 tests/claude-code：好的技能要能通过会话行为证明，而不是只看起来合理。
